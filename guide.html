VoiceOver
문제 정의

시각이 전제된 GUI는 화면을 볼 수 없는 사람에게는 본질적으로 닫혀 있다. 버튼이 어디 있는지, 무엇이 변했는지, 지금 무엇을 보고 있는지 — 모든 정보가 시각 채널로만 흐른다.

솔루션

화면의 모든 요소를 음성으로 읽어 주는 스크린 리더. 손가락 제스처(rotor, swipe), 키보드(VO+키), Bluetooth 점자 디스플레이, visionOS에서는 응시·핀치 제스처로 조작한다.

의미와 구조를 만들 때 .accessibilityLabel("Favorites")

텍스트 없이 아이콘·이미지만 보여 주는 모든 인터랙티브 요소. Button("Save")처럼 라벨이 들어 있는 표준 컴포넌트는 자동으로 잡힌다.

적용 전
하트 모양 아이콘만 있는 버튼은 VoiceOver가 "heart fill" 같은 시스템 아이콘 이름을 영어로 읽거나, 라벨이 없으면 그냥 침묵한다.
→
적용 후
포커스되는 순간 "좋아요"라고 또렷이 발화. Traits이 함께 있으면 "좋아요. 버튼."
상태를 전달할 때 .accessibilityValue(isLiked ? "On" : "Off")

탭 후 상태가 변하는 토글 버튼·체크박스·스위치 전체. 없으면 VoiceOver는 라벨만 반복하고 상태가 바뀐 사실을 알리지 않는다.

적용 전
"좋아요. 버튼." 이후 침묵. 좋아요가 눌린 건지 아닌지 알 수 없다.
→
적용 후
"좋아요. On. 버튼." / "좋아요. Off. 버튼." — 항상 상태를 함께 읽는다.
제스처 대체 .accessibilityAction(named: "Delete") { delete() }

스와이프 전용 인터랙션 전체 — 리스트 행 스와이프 삭제, 드래그 재정렬 등. VoiceOver 사용자는 같은 방식으로 스와이프 제스처를 수행할 수 없다.

적용 전
스와이프 삭제는 시각적으로 존재하지만 VoiceOver 사용자는 영원히 도달할 수 없다.
→
적용 후
두 손가락 스크럽 또는 Actions 로터에 "Delete"가 나타나 모든 입력 수단으로 동일 동작에 접근 가능.
활성화 전 결과 미리보기 .accessibilityHint("Opens navigation sidebar")

더블탭 결과가 라벨만으로 명확하지 않을 때만 사용한다. 힌트에 라벨을 다시 언급하지 않는다. VoiceOver는 힌트를 마지막에 읽으며 — 라벨·값·Trait 이후 — 사용자가 설정에서 전역으로 비활성화할 수 있다.

적용 전
"메뉴 버튼." — 더블탭이 사이드바를 여는데, 사용자는 실행하기 전에 이 사실을 알 방법이 없다.
→
적용 후
"메뉴 버튼. 탐색 사이드바를 엽니다." — 결과가 사전에 선언되어 시행착오가 필요 없다.
읽기 순서 재정의 .accessibilitySortPriority(1)

ZStack, 겹치는 뷰, 배지 오버레이, 시각적 읽기 순서가 뷰 계층 순서와 다른 2열 그리드. 높은 우선순위 숫자가 먼저 읽힌다. 자연스러운 DOM 순서가 명백히 잘못될 때만 사용한다.

적용 전
배지 오버레이가 있는 카드: VoiceOver가 "New"를 먼저 읽는다(ZStack의 마지막 = 히트 테스트 상단) 그런 다음 카드 제목을 읽는다 — 의미상 역순이다.
→
적용 후
카드는 `.accessibilitySortPriority(1)`, 배지는 `0`. VoiceOver가 카드 제목을 먼저 읽고 배지를 나중에 읽는다 — 시각 사용자가 레이아웃을 스캔하는 방식과 일치한다.
로터 — 요청 시 보조 메타데이터 .accessibilityCustomContent("Price", "$149")

가격·평점·거리·날짜처럼 여러 메타데이터 필드가 있어 매번 포커스 시 읽으면 소음이 될 항목이나 카드. 커스텀 콘텐츠는 사용자가 로터에서 "추가 콘텐츠" 옵션을 선택할 때만 나타난다.

적용 전
라벨: "Nike Air Max. 상품. ₩189,000. 별 4.5. 2.1km." — 50개 항목 목록을 스와이프할 때마다 전체를 읽는다. 피로하다.
→
적용 후
라벨: "Nike Air Max. 상품." 로터 → 추가 콘텐츠 → 가격: ₩189,000 / 평점: 별 4.5 / 거리: 2.1km — 요청 시 각각 개별로 표시.
복잡한 컨트롤에 프록시 뷰 .accessibilityRepresentation { Slider(value: $rating, in: 1...5) }

별점 선택기, 캔버스 슬라이더, 세그먼트 바, 원형 진행 링처럼 완전히 커스텀으로 그린 컨트롤 — 뷰 계층이 시맨틱 의미를 전혀 갖지 않는 경우. 프록시 뷰는 표시되지 않는다. VoiceOver가 커스텀 컨트롤을 해석하는 방식만 정의한다.

적용 전
커스텀 5점 별점 컨트롤: Shape 뷰 5개. VoiceOver가 각각 "circle"로 읽는다 — 역할 없음, 값 없음, 별점 변경 방법 없음.
→
적용 후
"별점: 3. 위아래로 밀어 조정하세요." — 전체 슬라이더 시맨틱이 적용된다. 시각적 변경 없음.
가상 더블탭 위치 이동 .accessibilityActivationPoint(CGPoint(x: frame.maxX - 20, y: frame.maxY - 20))

기하학적 중심이 의도한 타겟으로 탭을 전달하지 않는 뷰 — 모서리 동작 버튼이 있는 카드, 지도 주석 버블, 툴팁 오버레이. VoiceOver 더블탭은 요소 중심이 아닌 활성화 지점에서 발동된다.

적용 전
카드의 "열기" 버튼이 우하단 모서리에 있다. VoiceOver 더블탭이 중앙에서 발동 — 장식 이미지에 닿는다. 카드는 선택되지만 아무것도 열리지 않는다.
→
적용 후
활성화 지점을 버튼 위치로 설정하면 더블탭이 항상 "열기"에 닿는다 — VoiceOver 포커스 강조 사각형 위치와 무관하게.
명시적 포커스 탐색 순서 정의 view.accessibilityElements = [cell0, cell1, cell2, ...]

커스텀 컬렉션 뷰, 게임 보드, 캔버스 레이아웃, 자동 탐색이 잘못된 순서를 생성하는 모든 컨테이너. 컨테이너의 `accessibilityElements`를 VoiceOver가 방문해야 하는 자식 뷰의 순서 있는 배열로 설정한다.

적용 전
3×3 사진 그리드: VoiceOver가 Z 순서로 탐색 — 아래 행 먼저, 그다음 중간, 마지막으로 위. 완전히 직관적이지 않다.
→
적용 후
행 우선 순서로 셀을 설정: 좌상단 → 상단 중앙 → 우상단 → 중간 좌측… VoiceOver가 왼쪽에서 오른쪽, 위에서 아래로 탐색 — 시각적 읽기와 일치.
로터 — 이름 붙인 탐색 축 UIAccessibilityCustomRotor(name: "Links") { predicate in ... }

기사·링크·이미지·양식 필드·검색 결과처럼 반복되는 콘텐츠 유형이 있는 앱. VoiceOver 로터(두 손가락 회전 제스처)를 커스텀 축으로 확장할 수 있다. 사용자가 모든 요소를 하나씩 스와이프하지 않고 위아래 플릭으로 해당 유형 항목 간에 바로 이동한다.

적용 전
긴 기사 목록. VoiceOver 사용자가 제목을 찾기 위해 요소 하나씩 스와이프해야 한다 — 커스텀 레이아웃에는 "기사"나 "제목" 로터 축이 없다.
→
적용 후
`UIAccessibilityCustomRotor(name: "Articles")`에 `UIAccessibilityCustomRotorSearchPredicate`를 사용해 다음/이전 기사 셀을 반환한다. VoiceOver 로터에 "Articles"가 나타난다 — 사용자가 위아래 플릭으로 기사 간 바로 이동한다.
모달 — VoiceOver 포커스를 오버레이로 한정 modalView.accessibilityViewIsModal = true

기존 콘텐츠 위에 나타나는 커스텀 모달·바텀 시트·팝오버·오버레이 전체. 없으면 VoiceOver가 오버레이를 벗어나 시각적으로 접근 불가한 배경 요소에 포커스할 수 있다 — 치명적 결함이다.

적용 전
커스텀 바텀 시트가 메인 화면 위에 나타난다. VoiceOver 사용자가 시트의 마지막 요소를 지나 스와이프하면 오버레이 뒤에 시각적으로 숨겨진 배경 버튼에 포커스된다.
→
적용 후
`accessibilityViewIsModal = true`가 VoiceOver에게 이 뷰가 화면을 소유함을 알린다. 포커스가 시트 안에 갇힌다 — 배경 요소는 무시된다. 마지막 항목을 지나 스와이프하면 첫 번째 항목으로 돌아온다.
가상 요소 — 그려진 콘텐츠의 접근성 노드 UIAccessibilityElement(accessibilityContainer: self)

UIView 백킹이 없는 캔버스 UI·Core Graphics 콘텐츠·SpriteKit 씬·게임 보드·차트 데이터 포인트 전체. `UIAccessibilityElement`는 VoiceOver가 포커스할 수 있는 커스텀 프레임의 가상 접근성 노드를 생성한다.

적용 전
Core Graphics로 완전히 그린 막대 차트. VoiceOver가 UIView 하나만 본다 — 아무것도 유용하게 읽지 못한다. 모든 데이터 포인트에 접근할 수 없다.
→
적용 후
막대마다 `UIAccessibilityElement`를 하나씩 생성하고, 각각 `.accessibilityLabel`("January: $4,200"), `.accessibilityFrame`(화면 좌표의 막대 CGRect), `.accessibilityTraits = .staticText`를 설정한다. VoiceOver가 차트를 막대 단위로 탐색한다.
포커스 이벤트 — VoiceOver 포커스에 반응 override func accessibilityElementDidBecomeFocused()

포커스 시 스크롤하여 보이게 해야 하거나, 미리 듣기 소리를 재생하거나, 동반 뷰를 업데이트하거나, VoiceOver가 착지할 때 표시기를 애니메이션해야 하는 요소. `accessibilityElementDidBecomeFocused()`와 `accessibilityElementDidLoseFocus()` 모두 재정의한다.

적용 전
수평 캐러셀: VoiceOver가 화면 밖 카드에 포커스할 수 있지만 보이지 않는다 — 캐러셀이 포커스된 항목으로 스크롤되지 않는다. 사용자가 볼 수 없는 것의 라벨을 듣는다.
→
적용 후
`accessibilityElementDidBecomeFocused()`가 `scrollToItem(at:)`을 호출한다 — 캐러셀이 포커스된 카드로 스냅된다. 사용자는 항상 시각적으로 앞에 있는 것의 라벨을 듣는다.
음성 속성 — 스팬별 언어·IPA·피치 NSAttributedString(string: text, attributes: [.accessibilitySpeechLanguage: "ja-JP"])

혼합 언어 텍스트(일본어 제품명, 코드 식별자, 의학 용어), 약어의 커스텀 발음, 또는 대화에서 화자를 구분하기 위한 피치 변화. `accessibilityAttributedLabel`로 사용되는 `NSAttributedString` 범위에 속성을 적용한다.

적용 전
앱이 "WWDC"를 표시하면 VoiceOver가 글자 하나씩 "더블유 더블유 디 씨"로 읽는다. 제품명 "東京 Tokyo"는 기기의 영어 음성으로 한자를 읽는다 — 왜곡된다.
→
적용 후
한자 범위에 `.accessibilitySpeechLanguage: "ja-JP"`를 적용하면 일본어로 읽힌다. "WWDC"에 `UIAccessibilitySpeechAttributeIPANotation`을 적용하면 올바른 발음이 강제된다.
콘텐츠 유형 — TTS 읽기 스타일 최적화 .accessibilityTextContentType(.narrative)

합성기가 다른 억양을 적용해야 하는 본문: 산문에 `.narrative`, 단어 목록에 `.wordAlternative`, 비밀번호에 `.plain`, 코드 블록에 `.sourceCode`, 경로에 `.fileSystem`. 문장 부호 읽기 방식과 일시 중지 삽입에 영향을 준다.

적용 전
`Text` 뷰에 표시된 마크다운 소스. VoiceOver가 "**bold** and _italic_"을 읽는다 — 별표와 밑줄이 소리 내어 읽힌다. 결과는 읽을 수 없다.
→
적용 후
`accessibilityTextContentType(.sourceCode)`가 VoiceOver에게 코드임을 알린다. 문장 부호가 기호 이름으로 읽힌다("별표 별표 bold 별표 별표"). 코드가 탐색 가능하고 이해할 수 있게 된다.
경로 — 비직사각형 포커스 강조 element.accessibilityPath = UIBezierPath(ovalIn: bounds)

원형 버튼·파이 차트 조각·별점·지도 주석 형태처럼 VoiceOver가 그리는 직사각형 포커스 강조가 실제 인터랙티브 형태를 잘못 나타내는 요소 전체. `accessibilityPath`를 화면 좌표로 설정한다.

적용 전
원형 녹화 버튼: VoiceOver가 직사각형 포커스 박스를 그린다 — 모서리에 빈 공간이 보인다. 사용자가 원 주위의 직사각형을 보아 시각적으로 혼란스럽고 터치 타겟을 잘못 표현한다.
→
적용 후
`UIBezierPath(ovalIn: button.accessibilityFrame)`으로 VoiceOver가 버튼 형태에 정확히 일치하는 타원형 포커스 강조를 그린다. 포커스 윤곽이 정확하고 직관적이다.

문제 정의

개발자가 하드코딩한 폰트 크기는 사용자가 시스템에서 정한 크기를 무시한다. 저시력·고령·소글씨 피로 사용자 전체가 사실상 배제된다.

솔루션

Settings → Accessibility → Larger Text에서 선택한 크기에 따라 텍스트가 자동으로 확대·축소된다. 시스템이 12단계를 정의하며, 상위 5단계는 접근성 크기로 최대 약 310%까지 늘어난다.

자동 스케일 텍스트 .font(.body)

앱 안의 모든 텍스트. 시스템 시맨틱 스타일(.body, .headline, .caption)을 쓰면 자동으로 추적한다. 추가 코드 없음.

적용 전
고정 size: 17. 슬라이더를 AX5까지 올려도 텍스트는 여전히 17pt. 저시력 사용자의 설정이 완전히 무시된다.
→
적용 후
AX5에서 텍스트가 약 53pt까지 자동 확장. 필요 시 레이아웃이 단일 컬럼으로 전환. 사용자가 편안하게 읽을 수 있다.
커스텀 폰트 스케일 .font(.custom("Fraunces", size: 17, relativeTo: .body))

커스텀 브랜드 폰트. relativeTo: 없이 size만 쓰면 사용자 설정과 무관하게 17pt로 고정된다.

적용 전
.font(.custom("Fraunces", size: 17)) — 절대 스케일되지 않는다.
→
적용 후
relativeTo: .body가 커스텀 폰트를 body 단계에 고정시켜 시스템 텍스트와 동일 비율로 확대·축소된다.
여백 Dynamic Type 연동 @ScaledMetric(relativeTo: .body) var iconSize: CGFloat = 24

텍스트와 함께 커야 하는 커스텀 여백·아이콘 크기·패딩 상수. 없으면 텍스트가 AX 크기에 도달했을 때 고정 크기 컨테이너를 넘쳐 레이아웃이 깨진다.

적용 전
아이콘 프레임이 고정 Image(...).frame(width: 24, height: 24). AX5에서 인접 라벨은 53pt지만 아이콘은 24pt로 유지 — 비례적으로 작고 시각적으로 파괴된다.
→
적용 후
@ScaledMetric(relativeTo: .body) var iconSize: CGFloat = 24이 AX5에서 아이콘을 ~74pt로 확대해 기본 크기와 동일한 시각적 비율을 유지한다.
대형 텍스트 시 레이아웃 축소 ViewThatFits { HStack {...} VStack {...} }

텍스트가 매우 클 때 깨지는 나란히 배치 레이아웃(라벨+값 HStack). ViewThatFits는 각 자식을 순서대로 시도해 클리핑 없이 맞는 첫 번째를 사용한다.

적용 전
상품명 + 가격이 나란한 HStack. AX5에서 상품명이 "…"으로 잘린다 — 핵심 정보가 손실된다.
→
적용 후
ViewThatFits가 HStack을 먼저 시도하고 텍스트가 크면 VStack으로 폴백. 모든 Dynamic Type 크기에서 두 텍스트 모두 완전히 읽을 수 있다.
UIKit 스케일링 — 텍스트 크기에 맞춰 CGFloat 확대 UIFontMetrics.default.scaledValue(for: 24)

수동으로 크기를 지정한 여백 상수·아이콘 프레임·Dynamic Type에 맞춰 확대해야 하는 `CGFloat`이 있는 UIKit 코드. 앱 실행 중 텍스트 크기가 변경될 때 레이아웃을 업데이트하기 위해 `traitCollectionDidChange(_:)`에서 호출한다(iOS 17+: 재시작 없이 변경 가능).

적용 전
UIKit 셀의 `imageView.frame = CGRect(x: 0, y: 0, width: 44, height: 44)`. AX5에서 인접한 `UILabel`이 3줄로 줄 바꿈하지만 이미지는 44pt로 유지된다 — 비율적으로 깨진다.
→
적용 후
`traitCollectionDidChange`에서 `UIFontMetrics.default.scaledValue(for: 44)`가 이미지 프레임을 텍스트와 동기화하여 확대하도록 업데이트한다. 모든 Dynamic Type 크기에서 레이아웃 비율이 유지된다.

Reduce Motion
문제 정의

대부분 사용자에게 장식적인 애니메이션이 일부 사용자에게는 전정 장애 증상 — 어지러움, 구역질, 두통 — 을 유발할 수 있다. 취향 문제가 아니라 앱 자체를 물리적으로 사용할 수 없게 만든다.

솔루션

사용자의 모션 최소화 선호를 알리는 환경 변수. 이 값을 확인해 애니메이션을 게이팅 — 슬라이드 전환 제거, 패럴랙스를 불투명도로 대체, 자동 캐러셀 정지.

애니메이션 조건부 게이팅 @Environment(\.accessibilityReduceMotion) var reduce

직접 작성한 모든 애니메이션·전환. 시스템 제공 애니메이션(NavigationStack 슬라이드, 시트 표시)은 자동으로 게이팅된다 — 커스텀 애니메이션만 분기 필요.

적용 전
.animation(.spring, value: scale)이 무조건 실행 — 사용자가 Reduce Motion을 켠 상태에서도.
→
적용 후
.animation(reduce ? nil : .spring, value: scale) — 일반 사용자는 스프링, Reduce Motion 사용자는 즉각 전환.
UIKit Reduce Motion 상태 확인 UIAccessibility.isReduceMotionEnabled

UIView.animate 또는 CAAnimation으로 애니메이션을 추가하는 UIKit 뷰 컨트롤러 또는 UIView 서브클래스. 사용자가 설정을 토글할 때 동적으로 반응하려면 UIAccessibility.reduceMotionStatusDidChangeNotification을 등록한다.

적용 전
UIView.animate(withDuration: 0.4) { view.alpha = 1 } — Reduce Motion이 켜진 상태에서도 실행된다.
→
적용 후
guard !UIAccessibility.isReduceMotionEnabled else { view.alpha = 1; return } — reduce motion 활성화 시 애니메이션 없이 즉시 전환.
슬라이드 대신 크로스페이드 전환 .transition(reduceMotion ? .opacity : .slide)

내비게이션 전환, 탭 전환, 콘텐츠가 바뀌어야 하는 모든 콘텐츠 교체. 크로스페이드(opacity)는 전정 증상을 유발하지 않고 전환을 전달한다.

적용 전
탭 전환이 .slide를 사용 — 전체 화면이 전속력으로 옆에서 날아온다. 전정 장애 사용자가 메스꺼움을 보고한다.
→
적용 후
reduceMotion이 true이면 .opacity가 이전 콘텐츠를 페이드 아웃하고 새 콘텐츠를 페이드 인 — 움직임 없이 변화가 전달된다.
크로스페이드 — 시스템 권장 모션 대체 UIAccessibility.prefersCrossFadeTransitions

변화를 전달해야 하지만 슬라이딩이나 줌 애니메이션을 사용할 수 없는 내비게이션 전환과 탭 전환. `prefersCrossFadeTransitions`는 Reduce Motion이 켜져 있고 사용자가 명시적으로 크로스페이드를 대안으로 요청했을 때 true다 — `isReduceMotionEnabled`만보다 더 엄격한 신호다.

적용 전
앱이 `isReduceMotionEnabled`를 확인하고 모든 전환을 제거한다 — 뷰 변경이 시각적 신호 없이 즉각적이다. 일부 사용자가 전정 증상 없이도 급격한 전환이 방향 감각을 잃게 한다고 느낀다.
→
적용 후
`prefersCrossFadeTransitions`를 확인한다: true이면 `.opacity` 전환; `isReduceMotionEnabled`이지만 prefersCrossFade가 아니면 즉각 전환; 아니면 전체 스프링 사용. 세 수준의 모션 선호가 올바르게 존중된다.

Accessibility Traits
문제 정의

커스텀 뷰는 시맨틱 역할 정보를 갖지 않는다. 명시적 선언 없이 VoiceOver는 탭 가능한 카드를 정적 텍스트와 구별하지 못하고 "버튼"이나 "제목"으로 발화하지 않는다.

솔루션

Traits는 UIAccessibility의 시맨틱 어휘다. 올바른 Trait를 부착하면 표준 컴포넌트를 쓰지 않아도 VoiceOver가 버튼·제목·링크·이미지·선택됨을 정확히 발화한다.

인터랙티브 의도 선언 .accessibilityAddTraits(.isButton)

탭에 반응하지만 SwiftUI Button이나 UIButton이 아닌 모든 커스텀 뷰. Trait 없이는 VoiceOver가 "버튼"을 발화하지 않아 사용자가 인터랙티브 요소인지 알 수 없다.

적용 전
탭 가능한 커스텀 카드가 라벨만 읽힌다. 역할 힌트 없음. VoiceOver 사용자는 더블탭 해야 한다는 것을 알 수 없다.
→
적용 후
VoiceOver가 "기사. 버튼."을 발화한다. 사용자는 더블탭으로 활성화할 수 있음을 안다.
로터 제목 탐색 .accessibilityAddTraits(.isHeader)

스크롤 뷰의 섹션 제목. VoiceOver 로터에는 제목 모드가 있어 사용자가 위아래 플릭으로 섹션을 이동한다. Trait 없이는 제목이 로터에 나타나지 않는다.

적용 전
크게 스타일된 "최근 주문" 텍스트가 Headings 로터에 잡히지 않는다. 섹션으로 직접 이동할 수 없다.
→
적용 후
"최근 주문. 제목." — 로터에 나타나 스크롤 뷰 어디서든 바로 이동할 수 있다.
선택 상태 trait .accessibilityAddTraits(isChosen ? .isSelected : [])

커스텀 탭 바, 세그먼트 컨트롤, 필터 칩, 또는 표준 SwiftUI Picker가 아니지만 선택/비선택 상태를 가질 수 있는 요소. Trait이 설정되면 VoiceOver가 자동으로 "선택됨"을 발화한다.

적용 전
커스텀 탭 바 pill: "사진" 탭이 시각적으로 강조된다. VoiceOver가 "사진. 버튼."을 읽는다 — 이 탭이 현재 활성화됐다는 언급이 없다.
→
적용 후
활성 탭에 .isSelected trait이 있다. VoiceOver가 "사진. 선택됨. 버튼."을 읽는다 — 사용자가 더 탐색하지 않아도 어떤 탭이 현재인지 안다.
라이브 데이터 trait .accessibilityAddTraits(.updatesFrequently)

라이브 카운터, 주식 가격, 스포츠 점수, 또는 VoiceOver가 말하는 것보다 빠르게 변하는 숫자값. 이 힌트는 VoiceOver에게 모든 변경마다 요소를 다시 읽지 않도록 알려 끊임없는 방해 발화를 방지한다.

적용 전
라이브 입찰가가 2초마다 업데이트된다. VoiceOver가 각 업데이트를 읽으려 하고 계속 스스로를 방해한다 — 사용자가 부분 숫자들의 뒤섞인 흐름을 듣는다.
→
적용 후
.updatesFrequently가 VoiceOver에게 업데이트를 일괄 처리하라고 알린다. 사용자가 방해받는 대신 포커스를 이동해 요청 시 최신 값을 들을 수 있다.
원치 않는 시스템 trait 제거 .accessibilityRemoveTraits(.isButton)

부모 또는 시스템 컴포넌트에서 .isButton을 상속받지만 특정 맥락에서 순전히 정보 제공 목적인 뷰 — 예: 인터랙티브로 발화되지 않아야 하는 장식적 요약 타일로 렌더링된 버튼.

적용 전
Button 컨테이너 안에 내장된 읽기 전용 요약 카드가 "매출: $4,200. 버튼."을 읽는다 — 버튼 역할이 오해를 준다; 탭해도 의미 있는 동작이 없다.
→
적용 후
.accessibilityRemoveTraits(.isButton)이 역할을 제거한다. VoiceOver가 "매출: $4,200."을 읽는다 — 잘못된 상호작용 어포던스가 암시되지 않는다.
Adjustable — 스와이프 위아래로 값 변경 .accessibilityAddTraits(.isAdjustable)

값 범위를 나타내는 모든 커스텀 컨트롤 — 원형 다이얼·별점 피커·커스텀 슬라이더·볼륨 노브. `.isAdjustable`이 있으면 VoiceOver가 사용자에게 "위아래로 밀어 조정하세요"를 안내하고 해당 제스처에서 `accessibilityIncrement()`/`accessibilityDecrement()`를 호출한다.

적용 전
커스텀 5점 별점 뷰. VoiceOver가 "별점 3"을 발화하지만 상호작용 힌트가 없다. 사용자가 조정 가능한지 알지 못한다 — 정적 표시로 가정한다.
→
적용 후
.isAdjustable로 VoiceOver가 "별점 3. 조절 가능."을 발화하고 "위아래로 밀어 조정하세요"라는 암묵적 힌트를 제공한다. 사용자가 보이는 +/− 버튼 없이 별점을 조정한다.
직접 상호작용 — 원시 터치를 뷰에 전달 .accessibilityAddTraits(.allowsDirectInteraction)

피아노 건반·드로잉 캔버스·멀티터치 게임 컨트롤처럼 동시 여러 터치 포인트가 필요한 뷰. 이 Trait이 있으면 VoiceOver가 터치를 탐색 제스처로 처리하는 대신 실제 터치 이벤트를 뷰에 직접 전달한다.

적용 전
피아노 앱: VoiceOver가 모든 터치를 탐색 제스처로 가로챈다. 코드를 연주할 수 없다 — 모든 터치가 VoiceOver 스와이프가 된다. 악기가 완전히 조작 불가다.
→
적용 후
.allowsDirectInteraction이 VoiceOver에게 원시 터치를 피아노 뷰에 전달하라고 알린다. 사용자가 정상적으로 연주한다 — VoiceOver가 가로채지 않는다. 음악 연주가 완전히 접근 가능하다.
페이지 넘기기 — VoiceOver가 페이지 전환 콘텐츠로 처리 .accessibilityAddTraits(.causesPageTurn)

전자책 리더·만화 뷰어·문서 페이지네이터처럼 "다음" 동작이 페이지를 넘겨야 하는 앱. `UIAccessibilityReadingContent`와 결합하면 VoiceOver 자동 읽기 흐름과 통합된다 — Speak Screen이 마지막 요소에 도달하면 자동으로 페이지를 넘긴다.

적용 전
"다음 페이지" 버튼이 있는 전자책 리더. Speak Screen 연속 읽기 중 VoiceOver가 마지막 단어까지 읽고 멈춘다 — 페이지를 넘겨야 한다는 것을 알 방법이 없다.
→
적용 후
페이지 뷰에 `.causesPageTurn`을 설정하면 페이지 넘기기 동작이 콘텐츠를 전진시켜야 함을 신호한다. Speak Screen이 사용자 조작 없이 페이지를 넘어 끊김 없이 읽는다.

Element Grouping
문제 정의

아이콘·제목·부제목이 있는 카드 하나를 읽는 데 VoiceOver 스와이프 3번이 필요하다. 장식용 이미지는 무의미한 아이콘 이름을 발화해 소음을 더한다.

솔루션

`.accessibilityElement(children: .combine)`은 모든 자식 라벨을 하나의 발화로 합친다. `.accessibilityHidden(true)`은 장식 요소를 접근성 트리에서 완전히 제거한다.

카드 자식 요소 병합 .accessibilityElement(children: .combine)

단일 단위로 읽혀야 할 텍스트를 담은 리스트 행, 카드, HStack/VStack 전체. VoiceOver 정류장 3개가 1개로 줄어든다.

적용 전
아이콘("star.fill"), 제목("Pro Plan"), 부제목("$9.99/month") — VoiceOver 정류장 3개, 각각 제스처 필요.
→
적용 후
하나의 포커스로 "Pro Plan. $9.99/month."을 읽는다. 아이콘은 조용히 병합되거나 생략된다.
장식 요소 제거 .accessibilityHidden(true)

순전히 장식적인 이미지, 구분선, 배경 도형, 인접 텍스트와 내용이 중복되는 아이콘.

적용 전
배경 원이 "circle.fill"을 발화한다. 탐색 중 의미 없는 방해.
→
적용 후
원은 VoiceOver에서 보이지 않는다. 포커스가 바로 의미 있는 콘텐츠로 이동한다.
자식 보존 및 탐색 경계 설정 .accessibilityElement(children: .contain)

자식이 개별적으로 포커스 가능해야 하는 컨테이너(예: 여러 필드가 있는 양식 섹션)지만 VoiceOver가 그룹을 탐색 랜드마크로 인식해야 하는 경우. .combine과 달리 자식이 병합되지 않아 각자 포커스 정류장을 가지지만 컨테이너도 레이블된다.

적용 전
Settings 양식에 "계정" 섹션 헤더와 아래 4개 필드. VoiceOver가 섹션 개념이 없다 — 그룹화 맥락 없이 모든 필드를 평면 목록으로 읽는다.
→
적용 후
섹션 VStack에 .contain + .accessibilityLabel("계정 설정") + .accessibilityAddTraits(.isHeader)로 각 필드를 개별 접근 가능하게 유지하면서 로터 탐색이 섹션으로 직접 이동할 수 있다.
긴 페이지 건너뛰기 탐색 .accessibilityElement(children: .contain).accessibilityLabel("Main content")

적용 후

솔루션

환경에서 `accessibilityHighContrast`와 `accessibilityReduceTransparency`를 읽는다. 중간 톤 색상을 고대비 대안으로 교체하고 반투명 레이어를 불투명 배경으로 대체한다.

고대비 색상으로 교체 @Environment(\.accessibilityHighContrast) var highContrast

적용 전
.systemBackground` 위 `.secondary`(회색) 라벨 — 대비율 약 3.7:1. 비이상적 조건에서 WCAG AA 불합격.
→
적용 후

`Material` 또는 `UIBlurEffect`를 배경으로 사용하는 모든 뷰. 블러는 위에 겹친 텍스트의 유효 대비를 낮춘다.

적용 전
사이드바가 `Material.ultraThinMaterial`을 사용한다. 배경이 화려하면 텍스트 대비가 무너진다.
→
적용 후
`reduceTransparency`가 true이면 사이드바가 단색 `.systemBackground`로 전환된다. 텍스트가 항상 읽힌다.

적용 전

→
적용 후

Differentiate Without Color
문제 정의

남성의 약 8%는 색맹이다. 색상만으로 의미를 전달하는 UI 패턴 — 빨강 = 오류, 초록 = 성공 — 은 적록색맹 사용자에게 완전히 모호하다.

솔루션

환경에서 `accessibilityDifferentiateWithoutColor`를 읽어 색상 기반 지시자 옆에 아이콘·패턴·텍스트 레이블 같은 보조 단서를 추가한다. 색조 인식 없이도 의미가 전달된다.

색상 점에 형태 단서 추가 @Environment(\.accessibilityDifferentiateWithoutColor) var noDiffColor

색상이 유일한 신호인 모든 색상 점, 배지, 차트 세그먼트, 상태 표시기. 색상 변경만으로 표시되는 선택 상태 포함.

적용 전
네트워크 표시기: 초록 점 = 온라인, 빨강 점 = 오프라인. 적록색맹 사용자에게 두 점 모두 비슷한 황갈색으로 보인다.
→
적용 후
`noDiffColor`가 true이면 점이 "●"(온라인) 또는 "✕"(오프라인)으로 바뀐다. 형태가 색상 독립적으로 의미를 전달한다.
차트 패턴 채우기 추가 chartForegroundStyleScale(mapping: { category in noDiffColor ? AnyShapeStyle(patterns[category]) : AnyShapeStyle(colors[category]) })

색조만으로 세그먼트를 구별하는 모든 차트(막대, 원형, 선). noDiffColor가 true이면 해칭, 점, 대각선 줄무늬를 각 세그먼트에 추가해 색상뿐 아니라 질감으로도 의미를 전달한다.

적용 전
파이 차트: 빨강/초록/파랑으로 지역별 매출 분석. 적색맹 사용자는 구별할 방법 없이 비슷한 갈색 세그먼트 세 개를 본다.
→
적용 후
noDiffColor가 true이면 세그먼트가 고유한 패턴(단색/대각선 줄무늬/점)을 얻는다. 범례가 색상과 패턴을 모두 표시한다 — 완전한 색맹에서도 의미가 살아남는다.
SF Symbols 단색 vs 다색 Image(systemName: icon).symbolRenderingMode(noDiffColor ? .monochrome : .multicolor)

통과/실패 상태를 나타내는 데 사용되는 다색 SF Symbols(예: 초록색 checkmark.circle.fill, 빨간색 xmark.circle.fill). 단색 모드에서 모든 심볼이 틴트 색상으로 렌더링된다 — 상태가 색상이 아닌 심볼 모양으로 전달되어야 한다.

적용 전
초록색 checkmark.circle.fill과 빨간색 xmark.circle.fill이 적록색맹 사용자에게 구별되지 않는다 — 둘 다 동일한 황갈색으로 보인다.
→
적용 후
noDiffColor가 true이면 .monochrome + 고유한 형태로 전환: checkmark.circle(윤곽선) vs xmark.circle.fill(채워진). 형태만으로 통과/실패 의미가 전달된다.

Smart Invert — ignoresInvertColors
문제 정의

Smart Invert Colors는 저시력 사용자에게 인기 있는 도구로 화면 전체를 반전한다. 조치 없이는 사진·지도·영상도 반전되어 인식 불가능한 네거티브 필름 같은 결과물이 나온다.

솔루션

자연색으로 표시해야 하는 뷰에 `.accessibilityIgnoresInvertColors(true)`를 표시한다. 시스템이 해당 레이어를 반전 합성 단계에서 제외한다.

이미지 자연색 보존 .accessibilityIgnoresInvertColors(true)

사진·제품 사진·지도·차트·아바타·영상 미리보기를 표시하는 모든 Image. 앱 아이콘은 자동으로 제외되지만 그 외 이미지 콘텐츠는 아니다.

적용 전
제품 사진이 컬러 네거티브로 보인다. 흰색이 검정으로, 피부 톤이 반전된다. Smart Invert 상태에서 이미지를 알아볼 수 없다.
→
적용 후
사진이 자연색으로 렌더링된다. 주변 UI(내비게이션 바, 버튼)는 계속 반전을 따른다. 이미지 레이어만 제외된다.
UIKit 색상 반전 무시 imageView.accessibilityIgnoresInvertColors = true

UIKit UIImageView, WKWebView, MKMapView, AVPlayerLayer 컨테이너, 또는 사진·지도·영상을 표시하는 모든 UIView 서브클래스. SwiftUI의 .accessibilityIgnoresInvertColors(true)에 해당하는 UIKit 방법이다.

적용 전
제품 사진을 표시하는 UIImageView가 Smart Invert에서 반전된다 — 제품이 네거티브로 보인다. 고객이 색상이나 외관을 평가할 수 없다.
→
적용 후
imageView.accessibilityIgnoresInvertColors = true가 이미지 레이어를 반전 합성 단계에서 제외한다. 사진이 자연색을 유지하면서 주변 UI는 정상적으로 반전된다.
WKWebView CSS 색상 스키마 prefers-color-scheme: dark

사진·다이어그램·브랜드 에셋을 포함하는 WKWebView 콘텐츠. iOS의 Smart Invert가 다크와 유사한 환경을 트리거하므로 웹 콘텐츠는 prefers-color-scheme: dark 미디어 쿼리를 사용해 올바른 에셋을 제공하고 이중 반전 아티팩트를 방지해야 한다.

적용 전
WKWebView가 흰 배경 기사를 렌더링한다. Smart Invert가 다크로 반전하지만 내부의 모든 사진도 반전하여 이미 반전된 페이지 안에 네거티브 이미지를 생성한다.
→
적용 후
CSS @media (prefers-color-scheme: dark)가 다크 배경 스타일을 로드하고 필요한 경우 img 요소에 filter: none을 표시한다 — Smart Invert 상태와 무관하게 사진이 자연색을 유지한다.

문제 정의

버튼 탭 소리, 오류 알림음, 성공 핑처럼 소리만 사용하는 피드백은 청각장애 사용자와 시끄러운 환경에 있는 사용자에게 들리지 않는다. 동작 완료 확인이 전달되지 않는다.

솔루션

결과에는 `UINotificationFeedbackGenerator`, 물리적 느낌에는 `UIImpactFeedbackGenerator`를 사용해 모든 소리 이벤트에 햅틱을 함께 추가한다. 햅틱은 음성을 대체하는 것이 아니라 촉각 채널을 추가한다.

결과 햅틱 (성공 / 오류 / 경고) UINotificationFeedbackGenerator().notificationOccurred(.success)

양식 제출 확인, 결제 처리, 인증 성공 또는 실패. 사용자가 "됐다" 또는 "안 됐다"를 알아야 하는 모든 순간.

성공 시 부드러운 알림음이 재생된다. 시끄러운 카페에서 무음 모드이면, 사용자는 양식이 제출됐는지 불확실한 채 화면만 바라본다.
→
적용 후
.success`가 특유의 이중 탭 햅틱을 발생시킨다. 소리 없이 화면이 뒤집혀 있어도 사용자가 촉각으로 확인을 느낀다.
물리적 상호작용 느낌 UIImpactFeedbackGenerator(style: .medium).impactOccurred()

드래그 앤 드롭 착지, 슬라이더 스텝 스냅, 토글 켜짐, 재정렬 핸들 잡기. 가상 동작에 물리적 실재감을 부여한다.

적용 전
리스트 항목을 드래그해 놓아도 무게감이 없다. 드롭이 인식됐는지 확인이 없다.
→
적용 후
드롭 시 `.medium` 충격이 발생한다. 상호작용이 실재하는 느낌이 든다. 청각장애 사용자도 소리 대신 같은 촉각 신호를 받는다.
피커 스냅 햅틱 UISelectionFeedbackGenerator().selectionChanged()

스크롤 휠, 커스텀 피커, 세그먼트 슬라이더, 또는 사용자가 이산적 단계를 드래그하는 UI. 단계 변경마다 한 번 발동 — 최종 결과를 암시하지 않고 값이 변경됐음을 확인하는 특유의 디텐트 클릭 느낌을 생성한다.

적용 전
커스텀 드럼롤 날짜 피커가 소리 없이 스크롤된다. 떨림이 있는 사용자가 값이 새 날짜로 스냅됐을 때 느낄 수 없다 — 멈추고 디스플레이를 봐야 한다.
→
적용 후
selectionChanged()가 스크롤 중 각 날짜 스냅에서 발동된다. 촉각 디텐트 클릭으로 사용자가 보지 않고도 각 값 변경을 느낄 수 있어 기계식 피커의 물리적 느낌과 일치한다.
CoreHaptics 커스텀 햅틱 패턴 CHHapticEngine().makePlayer(with: CHHapticPattern(...))

표준 Apple 생성기와 다른 햅틱 서명이 필요한 브랜드 또는 게임플레이 상호작용 — 고유한 "잠금 해제" 패턴, 심박동 펄스, 모스 부호 스타일 알림. CoreHaptics가 강도, 선명도, 타이밍 곡선을 완전히 제어한다.

적용 전
앱이 모든 알림에 .success 햅틱을 사용한다 — 구매 확인, 메시지 수신, 레벨 완료가 모두 동일하게 느껴진다. 햅틱 피드백만 의존하는 사용자가 이벤트 유형을 구별할 수 없다.
→
적용 후
각 이벤트에 고유한 CHHapticPattern: 구매 = 단일 날카로운 탭, 메시지 = 이중 부드러운 펄스, 레벨 완료 = 강도 상승 램프. 촉각적 구분이 시각적 다양성과 일치한다.
watchOS 햅틱 — Taptic Engine 피드백 WKInterfaceDevice.current().play(.success)

watchOS 컴플리케이션·운동 마일스톤·타이머 완료·촉각 확인이 필요한 모든 watchOS 상호작용. Apple Watch Taptic Engine이 고유한 패턴을 제공한다: `.click`, `.success`, `.failure`, `.retry`, `.start`, `.stop`, `.notification`, `.directionUp/Down`.

적용 전
watchOS 운동 앱이 오디오만으로 "목표 달성!"을 발표한다. 무음 모드이거나 청각장애 사용자는 마일스톤 달성 피드백을 전혀 받지 못한다.
→
적용 후
`WKInterfaceDevice.current().play(.success)`가 Taptic Engine의 이중 탭 성공 패턴을 발동한다. 사용자가 손목을 통해 마일스톤 확인을 느낀다 — 소리가 필요 없다.

Voice Control — Input Labels
문제 정의

Voice Control은 사용자가 요소 이름을 말해 전체 UI를 조작하게 한다. 두 버튼의 라벨이 비슷하거나 버튼이 텍스트 없는 아이콘을 사용하면, Voice Control이 명령을 해석하지 못하거나 잘못된 요소를 활성화한다.

솔루션

`.accessibilityInputLabels(["보내기", "전송", "확인"])`은 여러 음성 별칭을 등록한다. `.accessibilityIdentifier("send-btn")`은 자동화와 그리드 모드 타겟팅을 위해 안정적인 프로그래밍 이름을 지정한다.

음성 별칭으로 구별 .accessibilityInputLabels(["Send", "Submit", "Go"])

표시 라벨이 모호하거나, 잘렸거나, 아이콘이거나, 기호인 버튼. 사용자가 시도할 법한 자연스러운 음성 대안을 제공한다.

적용 전
양식의 아이콘 버튼 "→"과 "✓". "화살표 탭" 또는 "체크 탭"을 말해도 아무 반응이 없다. Voice Control에 매핑이 없다.
→
적용 후
.accessibilityInputLabels(["다음", "확인"])` — "다음 탭" 또는 "확인 탭"을 말하면 매번 올바른 버튼이 활성화된다.
안정적인 자동화 이름 .accessibilityIdentifier("compose-send-btn")

자동화 스크립트나 UI 테스트의 타겟이 되는 핵심 인터랙티브 요소. identifier는 사용자에게 표시되지 않는다. 로컬라이제이션에 독립적인 기계 판독 가능 안정 이름이다.

적용 전
버튼 라벨이 로케일에 따라 변한다("Send" → "보내기"). 텍스트를 타겟으로 하는 자동화 스크립트가 로컬라이제이션에서 깨진다.
→
적용 후
`accessibilityIdentifier`는 로케일에 독립적이다. 스크립트와 테스트가 표시 텍스트가 아닌 ID를 참조한다.
반복 요소별 고유 레이블 .accessibilityLabel("Delete photo 1")

여러 항목이 동일한 동작 레이블을 공유하는 목록이나 그리드 — 행마다 반복되는 "삭제", "공유", "편집". Voice Control은 레이블이 모호할 때 숫자 오버레이를 표시하지만 고유한 레이블을 지정하면 구별 UI 없이 직접 음성 명령이 가능하다.

적용 전
각 썸네일에 "삭제" 버튼이 있는 사진 그리드. "삭제 탭"을 말하면 Voice Control이 모든 삭제 버튼에 숫자를 오버레이한다 — 사용자가 두 번째 단계로 "3 탭"을 말해야 한다.
→
적용 후
각 삭제 버튼이 사진 설명에서 .accessibilityLabel("해변 사진 삭제")를 얻는다. "해변 사진 삭제 탭"을 말하면 한 번의 명령으로 올바른 버튼이 활성화된다.
입력 레이블 — 다국어 음성 별칭 .accessibilityInputLabels(["Send", "Submit", "Go", "확인"])

사용자가 선호 언어로 명령을 말하는 이중 언어 또는 국제 앱. `.accessibilityInputLabels`는 여러 문자열을 허용하며 Voice Control이 그 중 하나와 일치시킨다. 영어와 로컬라이즈된 레이블을 모두 포함해 사용자가 어떤 언어를 말해도 앱이 동작하게 한다.

적용 전
이중 언어 앱: 버튼 레이블이 한국어로 "보내기". 영어 사용자가 "tap Send"를 말해도 한국어 레이블만 등록되어 있어 Voice Control이 영어 단어에 매핑이 없다.
→
적용 후
.accessibilityInputLabels(["보내기", "Send", "Submit"])이 세 가지를 모두 등록한다. 한국어나 영어를 말하는 사용자가 자신의 언어로 버튼을 활성화한다 — 하나의 구현으로 모든 로케일을 지원한다.

Full Keyboard Access & Focus Management
문제 정의

모달이나 알림이 프로그래밍적으로 나타날 때, 키보드와 스위치 제어 포커스가 트리거 요소에 남아 있다. 전면 콘텐츠에 도달할 수 없는 채 배경에 갇힌다.

솔루션

`@AccessibilityFocusState`를 사용해 새로 표시된 콘텐츠의 첫 번째 의미 있는 요소로 포커스를 프로그래밍적으로 이동한다. 닫을 때는 트리거 요소로 포커스를 복원한다.

새 콘텐츠 등장 시 포커스 이동 @AccessibilityFocusState var isConfirmFocused: Bool

시트, 알림, 인라인 유효성 검사 메시지, 동적으로 나타나는 모든 콘텐츠. 포커스를 이동하지 않으면 키보드·스위치 사용자가 배경에 갇혀 전면에 도달하기 위해 전체 화면을 탐색해야 한다.

적용 전
"보내기" 탭 → 확인 시트가 올라옴 → 포커스는 시트 뒤 "보내기"에 남아 있다. Tab 키가 배경 요소를 순환하며 시트에 절대 도달하지 못한다.
→
적용 후
.onAppear { isConfirmFocused = true }`가 즉시 포커스를 시트의 첫 번째 요소로 이동한다. 사용자가 "확인. 시트."를 듣고 즉시 상호작용할 수 있다.
닫은 후 포커스 복원 focusedField = .sendButton

모달, 시트, 팝오버를 닫은 후 포커스가 열었던 컨트롤로 돌아와야 한다. 뷰 계층 구조 상단으로 떨어지면 안 된다.

적용 전
시트 닫힘 → 포커스가 내비게이션 바 제목으로 떨어진다. 사용자가 원래 위치로 돌아가려면 전체 화면을 Tab으로 이동해야 한다.
→
적용 후
닫을 때 `@AccessibilityFocusState`가 포커스를 "보내기" 버튼으로 복원한다. 사용자가 정확히 이전 위치에서 계속한다.
UIKeyCommand 키보드 단축키 UIKeyCommand(input: "n", modifierFlags: .command, action: #selector(newItem))

iPad 또는 Mac Catalyst 앱의 자주 사용하는 동작. Full Keyboard Access 사용자와 파워 유저가 탐색을 우회하는 단축키의 혜택을 받는다. UIViewController.keyCommands에 키 명령을 추가한다 — ⌘을 길게 누르면 키보드 단축키 오버레이에 나타난다.

적용 전
새 메모 만들기: 메모 목록으로 이동(Tab×4) → 추가 버튼 활성화(Space). 매일 수십 개의 메모를 만드는 키보드 전용 사용자에게 이는 지치는 일이다.
→
적용 후
⌘N이 키 명령으로 등록된다. 앱 어디서든 누르면 새 메모가 즉시 만들어진다 — 탐색 0회. 단축키가 발견 가능성을 위해 ⌘ 오버레이에 나타난다.
커스텀 키 처리 override func pressesBegan(_ presses: Set<UIPress>, with event: UIPressesEvent?)

탐색이나 동작을 위해 특정 키(방향키, 스페이스 바, 이스케이프)에 응답해야 하는 커스텀 인터랙티브 뷰 — 게임 보드, 캔버스 편집기, 음악 시퀀서. 표준 컨트롤은 자동으로 처리하지만 커스텀 뷰는 pressesBegan과 pressesEnded를 재정의해야 한다.

적용 전
커스텀 그리드 퍼즐: 방향키가 아무것도 하지 않는다. Full Keyboard Access 사용자가 각 셀에 개별적으로 Tab을 사용해야 한다 — 9×9 그리드 탐색에 81번의 Tab 누름이 필요하다.
→
적용 후
pressesBegan이 방향키를 방향성 셀 이동에 매핑한다. 사용자가 정상 속도로 방향키로 퍼즐 그리드를 탐색한다 — 트랙패드를 사용하는 시각 사용자와 동일한 경험.
포커스 가이드 — 커스텀 레이아웃의 포커스 공백 연결 let guide = UIFocusGuide(); guide.preferredFocusEnvironments = [targetView]

방향 탐색이 도달 불가한 "섬"을 남기는 커스텀 레이아웃(tvOS, 키보드가 있는 iPad) — 행의 마지막 항목에서 →를 누르면 인접한 그룹 전체를 건너뛴다. `UIFocusGuide`는 화살표 키 포커스를 특정 타겟으로 리다이렉트하는 보이지 않는 포커스 가능 영역을 배치한다.

적용 전
2열 레이아웃: 왼쪽 열에 버튼 5개, 오른쪽 열이 다른 수직 오프셋에서 시작. 왼쪽 열 아래 버튼에서 →를 누르면 오른쪽 열에 도달하지 못한다 — 포커스가 내비게이션 바로 이동한다.
→
적용 후
`UIFocusGuide`가 열 사이의 공백에 부착되어 오른쪽 열 첫 번째 버튼으로 리다이렉트한다. → 키가 예상대로 열 간에 탐색한다 — 도달 불가한 포커스 섬이 없다.

Button Shapes
문제 정의

내비게이션 바의 텍스트 전용 버튼("완료", "취소", "편집")은 Button Shapes가 꺼져 있으면 정적 텍스트와 구별할 수 없다. 인지·저시력 사용자는 인터랙티브 텍스트와 장식 텍스트를 시각적 어포던스 없이 구별하지 못한다.

솔루션

환경에서 `accessibilityShowButtonShapes`를 읽어 커스텀 텍스트 전용 인터랙티브 요소에 배경, 테두리, 밑줄을 추가한다. 표준 SwiftUI 버튼은 자동으로 준수한다.

인터랙티브 경계 가시화 @Environment(\.accessibilityShowButtonShapes) var showShapes

커스텀 컴포넌트에서 버튼으로 렌더링된 텍스트: 탭 가능한 리스트 푸터, 인라인 "전체 보기" 링크, TapGesture가 있는 일반 Text. 표준 `Button` 뷰는 자동으로 처리된다.

적용 전
TapGesture가 있는 일반 Text로 렌더링된 "전체 보기 →". Button Shapes 켜짐 — 아무것도 변하지 않는다. 인접한 정적 텍스트와 정확히 동일하게 보인다.
→
적용 후
`showShapes`가 true이면 텍스트 뒤에 `RoundedRectangle` 테두리가 나타난다. 인터랙티브 경계가 명확하게 보이고 탭 가능하다.
UIKit 버튼 형태 활성화 확인 UIAccessibility.buttonShapesEnabled

커스텀 텍스트 전용 인터랙티브 뷰가 있는 UIKit 앱 — UITapGestureRecognizer가 있는 UILabel, 커스텀 셀 푸터, 인라인 "더보기" 링크. 동적으로 반응하려면 UIAccessibility.buttonShapesEnabledStatusDidChangeNotification을 등록한다.

적용 전
UILabel "비밀번호를 잊으셨나요?"에 UITapGestureRecognizer가 있다. Button Shapes가 켜지면 아무것도 변하지 않는다 — 주변 정적 텍스트와 동일하게 보인다.
→
적용 후
layoutSubviews에서 UIAccessibility.buttonShapesEnabled를 확인한다. true이면 레이블에 CALayer 테두리나 밑줄을 추가한다 — 인터랙티브 어포던스가 명확하게 보인다.
링크 밑줄 대안 Text("Privacy Policy").underline(showShapes)

테두리가 시각적으로 어색해 보이는 본문 텍스트 내 텍스트 전용 링크(이용약관, 도움말 기사, 각주 참조). 밑줄은 어떤 맥락에서도 링크로 읽히는 범용적이고 타이포그래피에 자연스러운 어포던스다.

적용 전
"계속하면 개인정보 처리방침에 동의하게 됩니다." — "개인정보 처리방침"이 탭 가능하지만 주변 본문 텍스트와 동일하게 보인다. 커스텀 Text + TapGesture이기 때문에 Button Shapes를 켜도 아무것도 변하지 않는다.
→
적용 후
showShapes가 true이면 .underline(true)이 "개인정보 처리방침"을 명확한 링크로 만든다. 브라우저 관례와 일치한다 — 설명이 필요 없다.

Accessibility Notifications
문제 정의

토스트 메시지, 로딩 상태, 인라인 유효성 오류처럼 동적으로 변하는 UI는 포커스가 다른 곳에 있으면 VoiceOver에 전달되지 않는다. 보이지 않는 곳이 변해도 사용자는 아무것도 듣지 못한다.

솔루션

`.announcement` 또는 `.layoutChanged` 알림을 전송해 포커스를 이동하지 않고 VoiceOver 발화 큐에 메시지를 직접 전달한다. 화면 전체가 교체될 때는 `.screenChanged`를 사용한다.

발화 큐로 메시지 전송 UIAccessibility.post(notification: .announcement, argument: "Saved")

사용자가 트리거했지만 볼 수 없는 모든 상태 변경 후: 양식 저장, 장바구니 추가, 업로드 완료, 오류 표시. 남용 금지 — 모든 Announcement는 현재 VoiceOver 읽기를 중단시킨다.

적용 전
"저장" 탭 → 스피너 표시 → "저장됨!" 배너 페이드인. VoiceOver 포커스는 여전히 저장 버튼에 있어 배너가 절대 발화되지 않는다.
→
적용 후
성공 시 `.announcement`에 "저장됨"을 전달한다. VoiceOver가 즉시 "저장됨"을 발화한다. 현재 위치를 잃지 않고 동작 완료를 안다.
새 콘텐츠로 포커스 이동 UIAccessibility.post(notification: .layoutChanged, argument: targetView)

새 콘텐츠 섹션이 인라인으로 삽입됐을 때(예: 검색 결과가 검색 바 아래에 나타나거나, 유효성 오류가 필드 아래에 나타남) 포커스가 첫 번째 새 요소로 이동해야 한다. 포커스를 지시하기 위해 대상 뷰를 argument로 전달한다.

적용 전
검색이 검색 바 아래에 삽입된 12개 결과를 반환한다. VoiceOver 포커스가 검색 바에 유지된다 — 사용자가 결과가 나타났음을 발견하려면 12회 이상 아래로 스와이프해야 한다.
→
적용 후
.layoutChanged에 첫 번째 결과 셀을 argument로 전달하면 포커스가 "결과 1: …"로 직접 이동한다 — 사용자가 즉시 새 콘텐츠로 방향을 잡는다.
전체 뷰 교체 알림 UIAccessibility.post(notification: .screenChanged, argument: firstElement)

탐색 푸시 없이 전체 가시 콘텐츠를 교체할 때 — 모달 유사 전환, 전체 화면 영상 전환, 마법사 단계 교체. .screenChanged가 VoiceOver에 새 "화면"이 활성화됐음을 신호하고 포커스를 firstElement로 이동한다.

적용 전
온보딩 마법사가 withAnimation opacity 전환으로 1단계 콘텐츠를 2단계로 교체한다. VoiceOver 포커스가 이제 보이지 않는 1단계 요소에 유지된다 — 이후 모든 스와이프가 보이지 않는 콘텐츠에 닿는다.
→
적용 후
전환 후 .screenChanged에 2단계 제목 뷰를 argument로 전달하면 VoiceOver 포커스가 "4단계 중 2단계: 플랜 선택"으로 이동한다 — 사용자가 즉시 새 맥락으로 방향을 잡는다.
스크롤 위치 피드백 UIAccessibility.post(notification: .pageScrolled, argument: "Page 3 of 7")

사용자가 스크롤 표시기를 볼 수 없는 커스텀 페이지형 스크롤 뷰, 캐러셀, 페이지네이션 목록. 각 페이지 변경 후 .pageScrolled를 전송해 VoiceOver가 현재 위치를 발화하도록 한다 — "어둠 속에서 스크롤"하는 혼란을 방지한다.

적용 전
수평 사진 캐러셀: VoiceOver 사용자가 스크롤 제스처로 스와이프한다. 3번 스와이프 후 캐러셀에서 얼마나 진행했는지, 사진이 얼마나 남았는지 전혀 모른다.
→
적용 후
각 페이지 변경이 "사진 3/7"과 함께 .pageScrolled를 발동한다. VoiceOver가 "7페이지 중 3페이지"를 발화한다 — 시각적 표시기 없이 사용자가 완전한 위치 맥락을 얻는다.
우선순위 — 대기열 vs 즉시 발화 알림 UIAccessibility.post(notification: .announcement, argument: NSAttributedString(string: msg, attributes: [.accessibilitySpeechQueueAnnouncement: true]))

시간에 민감한 발화 vs 낮은 우선순위 발화. 기본적으로 새 `.announcement`는 VoiceOver가 현재 읽는 것을 중단시킨다. `accessibilitySpeechQueueAnnouncement: true`를 설정하면 현재 발화가 완료된 후 대기열에 추가된다 — 읽기를 방해하지 않아야 하는 긴급하지 않은 상태 업데이트에 적합하다.

적용 전
자동 저장이 30초마다 알림을 발동한다 — "문서가 저장됐습니다."가 읽는 중에 30초마다 중간에 사용자를 방해한다. 매우 방해가 된다.
→
적용 후
`accessibilitySpeechQueueAnnouncement: true`가 VoiceOver가 현재 요소를 마칠 때까지 "저장됨" 메시지를 대기열에 추가한다. 사용자가 자연스러운 일시 정지 시점에 저장 확인을 듣는다 — 방해가 아니라.

Zoom & Magnification
문제 정의

시스템 Zoom은 전체 화면을 확대하지만, 클리핑이나 변환을 적용하는 커스텀 스크롤 뷰는 배율 변경 시 깨진다 — 이미지가 픽셀화되고, 레이아웃이 무너지거나 확대 영역이 잘못 잘린다.

솔루션

앱 내 확대·축소에는 `UIScrollViewDelegate.viewForZooming(in:)`을 구현하고 `minimumZoomScale` / `maximumZoomScale`을 설정한다. 텍스트 전용 콘텐츠에는 직접 줌 대신 Dynamic Type을 사용한다.

핀치 줌 스크롤뷰 델리게이트 scrollView.minimumZoomScale = 1; scrollView.maximumZoomScale = 5

이미지 상세 뷰, PDF 뷰어, 지도, 다이어그램처럼 픽셀 수준의 검사가 필요한 콘텐츠. 일반 텍스트 레이아웃에는 적용하지 말 것 — 그 경우 Dynamic Type이 올바른 도구다.

적용 전
다이어그램 이미지가 표시 크기로 고정되어 있다. 저시력 사용자가 세부 사항을 읽기 위해 확대할 수 없다. 시스템 Zoom이 고정 해상도 래스터를 흐리게 만든다.
→
적용 후
UIScrollView 컨테이너 안에서 핀치 제스처로 5배까지 확대한다. SVG/벡터 콘텐츠는 어떤 배율에서도 선명하게 유지된다.
핀치 제스처 경량 확대 UIPinchGestureRecognizer(target: self, action: #selector(handlePinch(_:)))

UIScrollView 서브클래스가 아니지만 핀치-투-줌이 필요한 뷰 — 커스텀 주석 오버레이, 데이터 시각화 캔버스, 문서 미리보기. 제스처의 스케일을 min/max 클램핑으로 view.transform에 적용한다.

적용 전
커스텀 지도 주석 레이어가 UIView에 그려진 CALayer다. 시스템 줌이 흐리게 만든다 — 앱 내 줌이 없고 벡터 경로가 픽셀화된다.
→
적용 후
UIPinchGestureRecognizer가 0.5×–5× 범위로 클램핑된 CGAffineTransform(scaleX: scale, y: scale)을 적용한다. 벡터 주석이 모든 스케일에서 선명하게 유지되고 gestureRecognizer.state == .ended에서 다시 그린다.
확대 수준 발화 func scrollViewDidZoom(_ scrollView: UIScrollView)

모든 UIScrollView 기반 줌 구현. 줌 제스처가 끝난 후 VoiceOver에 새 줌 수준을 발화해 확대를 위해 줌을 사용하는 시각장애 사용자가 현재 상태를 알 수 있도록 한다.

적용 전
사용자가 PDF를 3배로 확대한다. VoiceOver가 아무 말도 하지 않는다 — 새 줌 수준의 확인도 현재 배율의 표시도 없다.
→
적용 후
scrollViewDidZoom이 "300%로 확대"와 함께 .announcement를 전송한다. 각 줌 제스처 완료 후 VoiceOver가 "300%로 확대"를 발화한다.
프로그래밍 줌 — 관심 영역으로 이동 scrollView.zoom(to: targetRect, animated: true)

레이블된 관심 영역이 있는 다이어그램·회로도·지도. "Zoom to [region]" 버튼 또는 접근성 액션을 제공해 프로그래밍적으로 `zoom(to:animated:)`를 호출하면 VoiceOver와 키보드 사용자가 핀치 제스처 없이 특정 영역을 확대할 수 있다.

적용 전
회로 다이어그램: VoiceOver 사용자가 "Capacitor C3" 레이블에 포커스한다. C3의 세부 사항을 보기 위해 확대할 방법이 없다 — 정확한 터치 좌표 없이는 핀치 줌을 사용할 수 없다.
→
적용 후
커스텀 접근성 액션 "C3 확대"가 `scrollView.zoom(to: c3Frame, animated: true)`를 호출한다. VoiceOver 사용자가 Actions 로터를 통해 모든 컴포넌트의 줌을 트리거한다 — 핀치 필요 없음.

Speak Selection / In-App TTS
문제 정의

기사·전자책·설명서처럼 읽기 중심의 앱은 사용자가 묵독할 수 있다고 가정한다. 낮은 문해력·난독증·저시력 사용자는 텍스트를 소리로 듣는 것에서 큰 혜택을 받지만, Speak Selection은 사용자가 길게 누르는 방법을 알아야만 작동한다.

솔루션

앱 내에 `AVSpeechSynthesizer`로 구동되는 명시적인 "소리 내어 읽기" 버튼을 추가한다. 외부 제스처 없이 앱 내에서 속도·음정·언어·일시정지/재개를 제어한다.

기사 텍스트 읽기 AVSpeechSynthesizer().speak(AVSpeechUtterance(string: text))

기사 리더, 레시피 단계, 경로 안내, 양식 안내 — 시각 콘텐츠와 함께 오디오가 도움이 되는 모든 곳. Speak Selection에 의존하기보다 앱 내 TTS를 선호한다.

적용 전
소리 내어 읽기 옵션이 없는 긴 기사. VoiceOver로 읽을 수 있지만 한 번에 한 단락씩 스와이프해야 한다 — 흐름이 끊긴다.
→
적용 후
"▶ 듣기" 버튼이 전체 기사를 연속으로 읽는다. 일시정지/재개 및 재생 속도 컨트롤로 사용자가 자신의 속도에 맞게 따라갈 수 있다.
음성 언어 및 품질 선택 AVSpeechSynthesisVoice(language: "ko-KR")

언어가 콘텐츠와 일치해야 하는 모든 TTS 구현. 항상 음성 언어를 명시적으로 지정한다 — 기본 음성이 콘텐츠 언어와 일치하지 않을 수 있는 기기 로케일을 사용한다. 긴 읽기 세션에는 .enhanced 또는 .premium 품질 음성을 선호한다.

적용 전
앱이 음성 설정 없이 AVSpeechUtterance(string: text)를 사용해 한국어 기사 텍스트를 말한다. 영어로 설정된 기기에서 TTS 엔진이 영어 음성으로 한국어 텍스트를 읽는다 — 완전히 알아들을 수 없다.
→
적용 후
AVSpeechSynthesisVoice(language: "ko-KR")가 올바른 언어 음성을 강제한다. .enhanced 품질로 필터링하면 가능한 경우 고충실도 Siri 음성이 선택된다 — 장문 읽기에 더 자연스럽다.
단어별 강조 델리게이트 speechSynthesizer(_:willSpeakRangeOfSpeechString:utterance:)

전자책 리더, 자막, 텔레프롬프터 앱에서 현재 말하는 단어를 표시하면 난독증이나 읽기 어려움이 있는 사용자가 따라갈 수 있다. 델리게이트가 각 단어가 말해지기 전에 NSRange를 받는다.

적용 전
TTS가 단락을 읽는 동안 사용자가 정적 텍스트 벽을 바라본다. 난독증 사용자가 위치를 잃는다 — 오디오와 텍스트가 동기화되지 않는다.
→
적용 후
willSpeakRangeOfSpeechString이 어트리뷰트 문자열에서 현재 단어를 강조한다. 사용자가 단어별 노래방 스타일 강조를 본다 — 오디오와 시각 채널이 완벽하게 동기화된다.

Sound Recognition
문제 정의

청각장애·난청 사용자는 현관 벨, 아기 울음, 화재 경보기처럼 청각이 정상인 사용자가 당연히 듣는 환경 음향 이벤트를 놓친다. 순수하게 오디오로만 신호를 보내는 앱은 이 사용자에게 사실상 접근 불가다.

솔루션

Sound Analysis 프레임워크(`SNClassifySoundRequest`)를 사용해 중요한 오디오 이벤트를 감지하고 시각적으로 표시한다 — 화면 깜박임, 지속 배너 표시, 햅틱 발생. 시스템 Sound Recognition 기능을 보완한다.

오디오 분류 및 시각 표시 SNClassifyRequest() + SNAudioStreamAnalyzer

오디오 알림에 의존하는 앱 — 베이비 모니터, 반려동물 추적기, 홈 보안 — 에서 사용자가 오디오를 끄거나 청각장애일 수 있다. 모든 소리 알림에 시각적 동등 항목을 함께 제공한다.

적용 전
아기 울음 감지 → 앱이 알림음을 재생한다. 무음 모드이거나 사용자가 청각장애인 경우 알림을 완전히 놓친다.
→
적용 후
알림음 + 전체 화면 깜박임 + 알림 배너 + 햅틱이 동시에 발생한다. 청각장애 사용자가 세 가지 독립적인 감각 채널을 통해 반드시 감지할 수 있다.
다중 소리 분류 옵저버 try request.addObserver(self, forSoundIdentifier: SNClassifierIdentifier.siren)

여러 고유한 소리에 대해 사용자에게 경고해야 하는 안전 필수 앱 — 화재 경보기 감지, 베이비 모니터, 도어벨 앱. 각 SNClassifierIdentifier(.siren, .smokeSmokeAlarm, .glassBraking, .coughing)에 옵저버를 추가하고 각각 고유하게 처리한다.

적용 전
앱이 화재 경보기만 듣는다. 건물이 공습 사이렌으로 대피하는 상황에서 사용자가 경고를 받지 못한다 — 잠재적으로 생명을 위협하는 공백.
→
적용 후
.siren, .smokeSmokeAlarm, .fireAlarm에 대한 옵저버가 각각 고유한 시각적 경고와 햅틱 패턴을 트리거한다. 모든 중요한 소리 유형이 고유한 응답으로 처리된다.
신뢰도 임계값 필터링 func request(_ request: SNRequest, didProduce result: SNResult)

SNResultsObserving.request(_:didProduce:)를 구현할 때 result.classifications.first?.confidence(0.0–1.0)로 필터링한다. 유사한 주변 소음에서 오탐을 방지하기 위해 0.7 이상의 경우에만 경고를 트리거한다.

적용 전
화재 경보 앱이 모든 큰 소음에 전체 화면 경고를 발동한다. 요리 소리, 떨어진 냄비, 액션 영화를 트는 TV가 모두 경보를 트리거한다 — 경보 피로로 사용자가 실제 경보를 무시하게 된다.
→
적용 후
SNResultsObserving이 confidence > 0.7과 화재 경보기 특정 식별자로 필터링한다. 진짜 고신뢰도 감지만 경고를 트리거한다 — 오탐률이 극적으로 감소한다.
다중 레이블 — 동시 발생 소리 이벤트 감지 result.classifications.filter { $0.confidence > 0.5 }.map(\.identifier)

실제 환경에는 동시 소리가 자주 있다 — 누군가 문을 두드리는 동안 전화가 울린다. `SNClassificationResult.classifications`는 상위 하나가 아니라 모든 감지된 소리 클래스와 신뢰도 점수를 반환한다. 전체 배열을 처리해 중첩 이벤트를 동시에 처리한다.

적용 전
베이비 모니터 앱이 가장 신뢰도 높은 소리만 반환한다. 문이 쾅 닫히고 아기가 동시에 울면 더 큰 소리만 알림을 트리거한다 — 아기 울음이 조용히 놓친다.
→
적용 후
0.5 신뢰도 이상의 모든 분류를 필터링한다. "문 쾅 닫힘"과 "아기 울음" 모두 독립적인 알림을 동시에 트리거한다. 소리 중첩으로 인해 안전에 중요한 이벤트가 놓치지 않는다.

Live Captions / Speech Recognition
문제 정의

화상 통화, 라이브 이벤트, 팟캐스트처럼 실시간 음성 콘텐츠는 자막이 사전 생성되지 않으면 청각장애 사용자에게 접근 불가다. 실시간 대화에는 자막을 만들 대본이 없다.

솔루션

실시간으로 오디오를 전사하기 위해 `SFSpeechRecognizer`를 통합한다. 인식된 텍스트를 화면에 지속적으로 표시하는 오버레이에 전달한다. 시스템 Live Captions 기능이 전역으로 이 작업을 수행하지만, 앱 내 통합은 스타일 제어를 제공한다.

실시간 음성-텍스트 변환 SFSpeechAudioBufferRecognitionRequest + SFSpeechRecognizer

마이크 또는 네트워크에서 오디오를 스트리밍하는 모든 앱 — 화상 통화, 음성 메시지, 라이브 스트림. 음성 인식 권한을 미리 요청한다. 사용자가 명시적으로 허용해야 한다.

적용 전
음성 통화 앱: 오디오 전용. 청각장애 참여자가 대화를 따라갈 방법이 없다. 지연이 있고 스타일링이 없는 시스템 Live Captions에 의존해야 한다.
→
적용 후
영상 옆에 앱 내 실시간 전사 패널이 나타난다. 텍스트는 고대비 큰 글씨로 스타일링되고 30초간 유지되며 스크롤하여 이전 내용을 볼 수 있다.
권한 흐름 SFSpeechRecognizer.requestAuthorization { status in ... }

음성 인식을 통합하는 모든 앱. 인식 작업을 생성하기 전에 권한을 요청해야 하며 앱이 .denied와 .restricted 상태를 우아하게 처리해야 한다 — 수동 자막 입력을 제공하거나 설정으로 연결한다.

적용 전
앱이 권한 확인 없이 SFSpeechAudioBufferRecognitionRequest를 시작하려 한다. 권한이 거부되면 인식이 자동으로 실패한다 — 청각장애 사용자가 설명 없이 빈 자막 패널을 본다.
→
적용 후
requestAuthorization이 먼저 상태를 확인한다. .authorized → 인식 시작. .denied → UIApplication.openSettingsURLString 링크와 함께 "설정에서 음성 인식 활성화" 표시. .restricted → 수동 텍스트 입력 폴백 제공.
단어 수준 타이밍 동기화 result.bestTranscription.segments

현재 단어를 강조하거나 타임스탬프가 있는 롤링 전사를 표시하는 라이브 자막 디스플레이를 구축할 때. 각 SFTranscriptionSegment에는 timestamp, duration, substring이 있다 — 이를 사용해 자막 표시를 오디오 재생과 동기화한다.

적용 전
라이브 자막이 인식된 모든 텍스트를 단일 UILabel에 덤프한다. 레이블이 폭발적으로 업데이트된다 — 긴 침묵 후 텍스트 벽이 나타난다. 사용자가 자연스러운 말의 리듬을 따라갈 수 없다.
→
적용 후
SFTranscriptionSegment 타이밍 데이터가 각 단어를 자연스러운 발화 타임스탬프에 자막 디스플레이에 애니메이션으로 표시한다. 자막 출력이 자막 오버레이처럼 느껴진다 — 타이밍이 맞고 읽기 쉬우며 동기화된다.
작업 생명주기 — 새 컨텍스트를 위한 취소 후 재시작 recognitionTask?.cancel(); recognitionTask = recognizer.recognitionTask(with: request)

인식 정확도가 시간이 지남에 따라 저하되는 장시간 음성 세션(1시간 강의, 회의). Apple의 음성 인식은 기기와 서버 모두에서 약 1분의 세션 제한이 있다. 정확도를 높게 유지하기 위해 45–55초마다 선제적으로 작업을 취소하고 재시작한다.

적용 전
라이브 자막 세션이 20분 동안 실행된다. 60초 후 시스템이 조용히 전사를 중단한다 — 자막 패널이 멈춘다. 사용자가 자막이 작동한다고 생각하지만 나머지 19분 동안 아무것도 받지 못한다.
→
적용 후
타이머가 50초마다 `SFSpeechRecognitionTask`를 취소하고 새 `SFSpeechAudioBufferRecognitionRequest`로 재시작한다. 자막이 무기한 실행된다 — 세션 타임아웃 공백 없음.

Switch Control
문제 정의

스위치 제어 사용자는 하나 또는 두 개의 물리적 스위치로 탐색한다 — 탭 한 번으로 포커스 이동, 두 번째 탭으로 활성화. 비표준 인터랙션을 구현하는 커스텀 컨트롤은 접근성 활성화·증감 메서드를 재정의하지 않으면 조작할 수 없다.

솔루션

커스텀 탭 동작에 `accessibilityActivate()`를 재정의한다. 스텝퍼·슬라이더·캐러셀에 `accessibilityIncrement()`와 `accessibilityDecrement()`를 재정의한다. 동작이 처리됐음을 확인하려면 `true`를 반환한다.

스위치 선택 처리 override func accessibilityActivate() -> Bool

UITapGestureRecognizer에 반응하거나 자체 히트 테스트 로직을 가진 커스텀 UIView 서브클래스. 표준 SwiftUI Button과 UIButton은 이를 자동으로 처리한다.

적용 전
커스텀 카드 뷰가 UITapGestureRecognizer에 반응한다. 스위치 제어로 선택 → 활성화 → 아무 일도 일어나지 않는다. 접근성 시스템이 제스처 인식기를 발동하지 않는다.
→
적용 후
재정의가 `true`를 반환하고 탭 제스처와 동일한 액션 핸들러를 호출한다. 스위치 제어 활성화가 손가락 탭과 구별되지 않는다.
스텝퍼·슬라이더 증감 지원 override func accessibilityIncrement() { value = min(value + step, max) }

커스텀 스텝퍼, 원형 다이얼, 평점 컨트롤, 또는 커스텀 조절 가능 뷰. Switch Control과 VoiceOver 모두 .isAdjustable 요소에 포커스 후 사용자가 위로(증가) 또는 아래로(감소) 스와이프할 때 이 메서드를 사용해 값을 증감한다.

적용 전
커스텀 볼륨 다이얼이 드래그 제스처에 반응한다. Switch Control이 선택 → 위로 스와이프 → 아무것도 일어나지 않는다. 볼륨을 변경하는 유일한 방법은 Switch Control 사용자가 수행할 수 없는 드래그다.
→
적용 후
accessibilityIncrement()가 볼륨을 한 단계 높인다; accessibilityDecrement()가 낮춘다. Switch Control 위/아래 스와이프가 직접 볼륨 변경에 매핑된다 — 컨트롤이 완전히 조작 가능하다.
커스텀 스크롤 컨테이너 override func accessibilityScroll(_ direction: UIAccessibilityScrollDirection) -> Bool

UIScrollView를 상속받지 않는 커스텀 스크롤 컨테이너 — UIPanGestureRecognizer로 구동되는 UIView, CAScrollLayer, 또는 커스텀 SwiftUI DragGesture 스크롤. Switch Control이 콘텐츠를 페이지로 이동하기 위해 accessibilityScroll을 사용한다.

적용 전
DragGestureRecognizer로 구동되는 커스텀 수평 카드 캐러셀. Switch Control 사용자가 카드를 스와이프할 수 없다 — 제스처 인식기가 접근성 스크롤 이벤트를 무시한다.
→
적용 후
accessibilityScroll(.right)가 다음 카드로 이동; .left가 뒤로 간다. "카드 2/5"를 발화하는 .pageScrolled 알림과 함께 true를 반환한다 — Switch Control 사용자가 캐러셀을 정상적으로 탐색한다.
커스텀 로터 — 콘텐츠 내 점프 탐색 accessibilityCustomRotors = [UIAccessibilityCustomRotor(name: "Paragraphs") { pred in ... }]

기사·문서·채팅 스레드·코드 편집기처럼 Switch Control과 VoiceOver 사용자가 논리적 단위(단락·메시지·함수·섹션)로 점프해야 하는 장문 콘텐츠 뷰. 커스텀 로터가 VoiceOver 로터에서 글자·단어·줄 같은 시스템 기본 로터와 함께 나타난다.

적용 전
200개 메시지가 있는 채팅 앱: Switch Control 사용자가 이전 메시지에 도달하기 위해 모든 요소(타임스탬프·아바타·메시지 텍스트·반응)를 방문해야 한다. 대화를 다시 읽는 데 수분이 걸린다.
→
적용 후
`UIAccessibilityCustomRotor(name: "Messages")`가 연속된 메시지 버블을 반환한다. Switch Control 로터를 "Messages"로 설정하고 위로 플릭하면 타임스탬프·아바타·반응을 완전히 건너뛰고 이전 메시지로 바로 이동한다.

Touch Target Size
문제 정의

44×44pt 미만의 아이콘 전용 버튼, 인라인 링크, 탭 바 항목 같은 작은 인터랙티브 타겟은 떨림·운동 장애 또는 손가락이 굵은 사용자에게 명중이 불균형하게 어렵다. 빗나가면 의도치 않은 활성화와 좌절감을 유발한다.

솔루션

시각적 크기와 히트 영역을 독립적으로 유지한다. UIKit에서는 `.contentEdgeInsets`, SwiftUI에서는 투명 오버레이를 사용해 시각 디자인을 변경하지 않고 탭 가능한 영역을 최소 44×44pt로 확장한다.

시각 변경 없이 히트 영역 확장 .frame(width: 44, height: 44).contentShape(Rectangle())

시각적으로 44×44pt 미만인 모든 탭 가능 요소 — 아이콘, 닫기 버튼("✕"), 접기 화살표, 세그먼트 항목. Apple HIG는 모든 인터랙티브 요소에 최소 44×44pt를 요구한다.

적용 전
닫기 버튼이 20×20 "✕" 아이콘이다. 운동 장애 사용자가 작은 영역에 정확히 탭해야 한다. 반복적인 빗나감이 잘못된 뷰를 닫거나 아무 일도 일어나지 않게 한다.
→
적용 후
아이콘은 시각적으로 20×20으로 유지된다. `.frame(width: 44, height: 44).contentShape(Rectangle())`이 전체 44pt 사각형을 탭 가능하게 만든다. 시각적으로 동일하고 기능적으로 접근 가능하다.
contentInsets로 히트 영역 확장 button.configuration?.contentInsets = NSDirectionalEdgeInsets(top: 12, leading: 12, bottom: 12, trailing: 12)

시각적으로 작은 UIKit UIButton 인스턴스 — 툴바의 아이콘 버튼, 내비게이션 바 항목, 인라인 동작 버튼. contentEdgeInsets(iOS 15+ 폐기) 또는 UIButton.Configuration.contentInsets가 시각적 크기를 변경하지 않고 탭 가능한 영역을 확장한다.

적용 전
툴바 아이콘 버튼: 24×24pt 시각적 크기, 24×24pt 히트 영역. 떨림이 있는 사용자가 자주 빗나가 인접 버튼을 트리거한다.
→
적용 후
configuration.contentInsets = .init(top: 10, leading: 10, bottom: 10, trailing: 10)이 아이콘을 24×24pt로 유지하면서 히트 영역을 44×44pt로 만든다. 인접 버튼은 영향을 받지 않는다.
불규칙 형태 히트 테스트 재정의 override func point(inside point: CGPoint, with event: UIEvent?) -> Bool

원형 버튼, 다이아몬드형 게임 타겟, 지도 핀처럼 경계 rect는 크지만 일부만 탭에 반응해야 하는 커스텀 형태의 인터랙티브 뷰. point(inside:with:)를 재정의해 확장된 또는 제한된 히트 영역을 프로그래밍적으로 정의한다.

적용 전
원형 재생 버튼: 36×36pt 뷰 안에 그려진 36pt 지름. 경계 rect의 모서리 영역이 투명하지만 탭 가능하다 — 사용자가 근처를 탭해 실수로 버튼을 트리거한다.
→
적용 후
override func point(inside point: CGPoint, with event: UIEvent?) -> Bool { return bounds.insetBy(dx: -4, dy: -4).contains(point) } — 모든 면에서 히트 영역을 4pt 확장해 원 밖의 히트를 제한하면서 순 44×44pt 타겟을 만든다.

Pointer Interaction
문제 정의

터치 대신 트랙패드·마우스·헤드 포인터에 의존하는 사용자(운동 장애, 키보드 중심 워크플로)는 커스텀 뷰에서 호버 피드백을 받지 못한다 — 버튼이 올라가지 않고, 카드가 강조되지 않으며, 커서 모양이 변하지 않는다.

솔루션

UIKit 뷰에 `UIPointerInteraction`을 추가하거나 SwiftUI에서 `.hoverEffect()` / `.pointerStyle()`을 사용한다. 탭 없이 인터랙티브성을 전달하기 위해 커서를 올리거나 강조하거나 모양을 바꾸는 포인터 영역을 정의한다.

인터랙티브 카드 리프트 효과 .hoverEffect(.lift)

iPad 또는 Mac Catalyst 앱의 모든 탭 가능한 카드, 리스트 행, 아이콘 버튼. `.highlight` 효과는 인라인 링크에, `.lift`는 카드 표면에 적합하다.

적용 전
트랙패드 사용자가 카드 그리드 위로 커서를 이동 — 커서가 기본 화살표로 유지되고 시각적 반응이 없다. 각각 시도하지 않으면 어떤 카드가 탭 가능한지 알 수 없다.
→
적용 후
카드를 호버하면 미묘한 그림자와 함께 올라오고 커서가 포인터 손 모양으로 바뀐다. 사용자가 탭 없이 즉시 인터랙티브임을 안다.
커스텀 포인터 영역 형태 UIPointerRegion(rect: button.bounds.insetBy(dx: -4, dy: -4), identifier: "playBtn")

뷰의 경계와 다른 포인터 상호작용 영역이 필요한 뷰 — 비직사각형 컨트롤, 오버플로 히트 영역, 또는 시각적 요소가 상호작용 영역보다 작은 뷰. UIPointerRegion이 효과를 트리거하는 정확한 rect를 뷰의 좌표 공간으로 정의한다.

적용 전
44pt 탭 타겟 안에 20pt 아이콘이 있는 아이콘 버튼. 44pt 영역 어디서나 호버하면 전체 44pt 영역에 .lift 효과가 활성화된다 — 시각적 리프트가 20pt 시각 버튼과 일치하지 않는다.
→
적용 후
UIPointerRegion(rect: iconBounds)가 리프트 효과를 20pt 아이콘 영역으로 제한한다. 호버 효과가 시각 버튼과 정확히 일치한다 — 포인터 상호작용이 물리적으로 정확하게 느껴진다.
인라인 텍스트 링크 강조 UIPointerStyle(effect: .highlight(UITargetedPreview(view: linkLabel)))

본문 텍스트 내 인라인 텍스트 링크. .highlight 효과(.lift와 달리)는 요소를 들어올리지 않고 배경을 어둡게 하고 둥글게 만든다 — 카드 유사 어포던스가 없는 텍스트 수준 요소에 적합하다.

적용 전
기사 본문 안의 인라인 "더 읽기 →" 텍스트. 호버하면 화살표 커서가 표시된다 — 시각적 반응 없음. 사용자가 클릭하기 전까지 텍스트가 인터랙티브한지 알 수 없다.
→
적용 후
.highlight 효과가 호버 시 링크 텍스트 주위에 둥근 강조를 생성한다. 커서가 포인터 손 모양으로 바뀐다. 링크가 인터랙티브로 시각적으로 구별된다 — 브라우저 링크 동작과 동일하다.
영역 델리게이트 — 호버 위치별 동적 포인터 영역 func pointerInteraction(_ interaction: UIPointerInteraction, regionFor request: UIPointerRegionRequest, defaultRegion: UIPointerRegion) -> UIPointerRegion?

각각 다른 포인터 스타일이 필요한 여러 인터랙티브 하위 영역이 있는 뷰 — 본문(highlight)과 액션 버튼(lift)이 있는 카드, 또는 왼쪽은 텍스트(기본 커서)이고 오른쪽은 삭제 버튼(lift + highlight)인 테이블 셀. 터치 위치에 따라 다른 영역을 반환하도록 델리게이트를 구현한다.

적용 전
전체 카드가 단일 `.lift` 효과를 사용한다. "삭제" 버튼을 호버하면 전체 카드가 올라온다 — 시각 효과가 오해를 준다(삭제 대상이 버튼만이 아닌 전체 카드처럼 보인다).
→
적용 후
델리게이트가 텍스트 영역 호버 시 영역 A(카드 본문, `.highlight`)를, 버튼 위 호버 시 영역 B(삭제 버튼 경계, `.lift`)를 반환한다. 각 하위 영역이 적절하고 정직한 포인터 효과를 갖는다.

Personal Voice
문제 정의

ALS·파킨슨병·진행성 언어 장애가 있는 사용자는 종종 언어 능력을 잃기 전에 Personal Voice 클론을 만든다. 일반 TTS 음성만 사용하는 앱은 사용자의 보존된 디지털 음성 정체성을 무시한다.

솔루션

AVSpeechSynthesizer.requestPersonalVoiceAuthorization으로 권한을 요청하고 AVSpeechSynthesisVoice.speechVoices()에서 `.personalVoice` 품질을 필터링한다. 발견된 음성을 AVSpeechUtterance에 사용해 앱이 사용자 자신의 음성으로 말하게 한다.

사용자 Personal Voice 클론 사용 AVSpeechSynthesizer.requestPersonalVoiceAuthorization { _ in }

텍스트 음성 변환 또는 커뮤니케이션 기능이 있는 모든 앱 — AAC 앱, 음성 메시지 작성기, 내비게이션 앱. Personal Voice를 열거하기 전에 항상 권한을 요청한다.

적용 전
커뮤니케이션 앱이 항상 일반 Siri 음성으로 말한다. Personal Voice를 훈련한 ALS 사용자가 자신을 대표하는 낯선 목소리를 듣는다 — 깊이 개인적인 실패다.
→
적용 후
권한 부여 후 앱이 `.personalVoice` 품질 음성을 찾아 사용한다. 사용자의 말이 자신의 보존된 목소리로 전해진다 — 존엄성과 정체성이 유지된다.
Personal Voice 열거 및 선택 AVSpeechSynthesisVoice.speechVoices().first { $0.voiceTraits.contains(.isPersonalVoice) }

requestPersonalVoiceAuthorization에서 .authorized를 받은 후 사용 가능한 음성을 열거하고 .isPersonalVoice trait이 있는 음성을 선택한다. 재사용을 위해 음성 식별자를 저장한다 — 매번 발화마다 다시 열거하는 것은 비용이 크다.

적용 전
앱이 requestPersonalVoiceAuthorization을 호출하지만 음성을 필터링하지 않는다 — 여전히 기본 Siri 음성을 사용한다. 권한 요청이 자신의 목소리를 기대했던 사용자를 혼란스럽게 했다.
→
적용 후
.authorized 시 .isPersonalVoice로 speechVoices()를 필터링한다. utterance.voice를 찾은 음성으로 설정한다. Personal Voice를 찾지 못하면(아직 훈련되지 않음) 설정 → 손쉬운 사용 → Personal Voice로의 앱 내 안내를 표시한다.
폴백 체인 구성 utterance.voice = personalVoice ?? enhancedVoice ?? AVSpeechSynthesisVoice(language: locale)

Personal Voice가 설정되지 않을 수 있는 프로덕션 TTS 기능. 항상 폴백 체인을 구축한다: personal voice → 콘텐츠 언어에 맞는 enhanced 품질 시스템 음성 → 기본 시스템 음성. nil 음성으로 발화가 자동으로 실패하도록 두지 않는다.

적용 전
앱이 utterance.voice = personalVoice를 설정한다 — 사용자가 Personal Voice를 훈련하지 않았으면 이는 nil이다. AVSpeechSynthesizer가 예측 불가능한 폴백 음성을 사용하는데 종종 잘못된 언어다.
→
적용 후
명시적 폴백 체인이 콘텐츠 언어에 사용 가능한 최선의 음성을 보장한다. Personal Voice를 훈련하지 않은 사용자가 enhanced Siri 음성을 얻는다 — 여전히 고품질, 올바른 언어, Personal Voice 설정 안내 포함.
음성 품질 — 장문 음성에 프리미엄 선호 AVSpeechSynthesisVoice.speechVoices().filter { $0.quality == .premium }

장문 읽기·내비게이션 턴-바이-턴·사용자가 장시간 듣는 모든 맥락. 프리미엄 음성(별도 다운로드, ~150MB)은 기본 번들 음성보다 훨씬 자연스럽다. 가용성을 확인하고 없으면 사용자에게 다운로드를 안내한다.

적용 전
팟캐스트 스타일 기사 리더가 기본 시스템 음성을 사용한다. 프리미엄 음성이 다운로드되지 않은 기기에서 TTS가 기계적으로 들린다 — 사용자가 기능을 비활성화하고 사용하지 않는다.
→
적용 후
앱이 `.premium` 품질 음성을 확인한다. 없으면 "설정 → 손쉬운 사용 → 말하기 콘텐츠 → 음성에서 향상된 음성을 다운로드하면 더 자연스러운 읽기 경험을 제공합니다."를 표시한다. 활성화한 사용자가 기능이 "실제로 즐거웠다"고 보고한다.

Live Speech / AAC Integration
문제 정의

보완대체의사소통(AAC) 사용자는 말하고 싶은 것을 입력해 음성으로 출력한다. 채팅 버블·양식처럼 텍스트 기반 출력만 제공하는 앱은 대면·전화 상호작용에서 음성 합성으로 소통하는 사용자를 배제한다.

솔루션

메시지 작성 뷰에 "말하기" 동작을 추가한다. 시스템 Live Speech(설정 → 손쉬운 사용 → 라이브 스피치)가 전역 케이스를 처리하지만, 앱 내 TTS는 앱을 전환하지 않고도 AAC 사용자에게 더 빠르고 맥락에 맞는 경로를 제공한다.

작성 후 한 탭으로 음성 출력 AVSpeechSynthesizer().speak(utterance)

채팅 앱, 메모 앱, 커뮤니케이션 도구 또는 실제 대화 맥락에서 사용되는 모든 텍스트 작성 표면 — 고객 서비스 키오스크, 의료 접수 양식, 음식점 주문.

적용 전
채팅 앱: 메시지 입력 → 텍스트로 전송. 카페 카운터의 AAC 사용자가 주문을 음성으로 전달하기 위해 별도의 AAC 앱으로 전환해야 한다 — 느리고 눈에 띈다.
→
적용 후
"🔊 말하기" 버튼이 전송 버튼 옆에 있어 입력한 텍스트를 스피커로 출력한다. 사용자가 동일한 앱에 머물며 상호작용이 원활하고 위엄 있다.
사전 로드 빠른 응답 라이브러리 let phrases = ["Yes", "No", "Thank you", "Please repeat that", "One moment"]

AAC 지향 앱 또는 속도가 중요한 모든 커뮤니케이션 기능. 일반적인 구문을 AVSpeechUtterance 객체로 미리 로드하고 speak()를 즉시 호출하면 TTS 초기화 지연을 피한다 — 50ms 응답과 600ms 지연의 차이는 대면 대화에서 중요하다.

적용 전
사용자가 모든 응답을 타이핑해야 한다. 상점에서 빠른 대화에서 타이핑 지연으로 상대방이 사용자가 끝났다고 가정하고 문장 중간에 자리를 뜬다.
→
적용 후
빠른 구문 바가 키보드 위에 8개의 구성 가능한 원탭 구문을 표시한다. "잠깐만요"를 탭하면 미리 생성된 AVSpeechUtterance가 즉시 발동된다 — 시간 민감한 교환에서 타이핑 지연 없음.
말하기 표시기 델리게이트 speechSynthesizer(_:didStart:) and speechSynthesizer(_:didFinish:)

모든 앱 내 TTS 기능. 합성이 활성화되는 동안 눈에 보이는 "말하는 중..." 표시기를 표시해 사용자(와 주변 사람들)가 오디오 출력이 진행 중임을 안다. didFinish에서 숨긴다. 이는 음성이 완료됐을 때 신호하기 위해 시스템에 의존하는 시청각장애 사용자에게 중요하다.

적용 전
사용자가 "말하기"를 탭한다. 합성기가 시작된다 — 하지만 시각적 표시기가 없다. 사용자가 음성이 끝났는지, 들렸는지, 무언가 잘못됐는지 알 수 없다.
→
적용 후
didStart가 애니메이션 파형 아이콘을 표시한다. didFinish가 숨기고 선택적으로 VoiceOver 사용자를 위한 .announcement 알림("완료")을 전송한다. TTS 생명주기가 완전히 보인다.

Guided Access
문제 정의

인지·발달 장애는 과제에 집중하기 어렵게 만들 수 있다. 방해 요소 없는 키오스크 또는 단일 목적 모드가 실수로 다른 화면으로 이동하는 것을 방지한다. 보호자와 교육자도 특정 앱이나 영역으로 기기를 잠글 필요가 있다.

솔루션

모드를 감지하고 UI를 그에 맞게 단순화하기 위해 `UIAccessibility.isGuidedAccessEnabled`를 확인한다. 키오스크 배포에서 프로그래밍적으로 진입/종료하기 위해 `UIAccessibilityRequestGuidedAccessSession`을 사용한다.

Guided Access 시 UI 단순화 UIAccessibility.isGuidedAccessEnabled

교육 앱, 엔터프라이즈 키오스크 앱, 치료 도구, 통제된 환경에서 배포되는 모든 단일 목적 앱. 동적으로 반응하려면 `UIAccessibility.guidedAccessStatusDidChangeNotification`을 등록한다.

적용 전
Guided Access 상태의 교실 앱이 여전히 설정 기어 아이콘을 표시한다. 학생이 탭 → 잠긴 세션 내에서 설정 UI가 나타난다. 보호자의 의도가 우회된다.
→
적용 후
isGuidedAccessEnabled가 true이면 설정 버튼이 숨겨지고 고급 탐색 옵션이 사라진다. 핵심 과제 UI만 남는다.
키오스크 잠금 프로그래밍 세션 제어 UIAccessibilityRequestGuidedAccessSession(true) { success in ... }

시작 시 자동으로 Guided Access에 진입해 사용자가 앱을 떠나지 못하도록 해야 하는 키오스크 앱(소매, 의료, 박물관). 기기에 Guided Access 패스코드가 설정되어 있어야 한다. 완료 핸들러가 세션이 성공적으로 시작됐는지 나타낸다.

적용 전
박물관 전시 iPad: 직원이 방문객에게 기기를 건네기 전에 트리플 클릭으로 Guided Access를 수동으로 활성화해야 한다. 잊으면 방문객이 전시 앱을 종료하고 기기에 접근할 수 있다.
→
적용 후
UIAccessibilityRequestGuidedAccessSession(true)이 applicationDidBecomeActive에서 호출된다. 전시 앱이 모든 시작 시 스스로 잠긴다 — 직원 조치 불필요. 방문객이 앱을 떠날 수 없다.
커스텀 기능 게이팅 제한 UIAccessibilityGuidedAccessRestrictionDelegate.accessibilityGuidedAccessRestriction(withIdentifier:)

Guided Access 세션 중 선택적으로 비활성화해야 하는 기능이 있는 앱 — 터치 입력, 특정 탭, 키보드 입력, 외부 URL 링크. 델리게이트를 구현해 내장된 터치 및 모션 제한과 함께 Guided Access 설정 UI에 나타나는 명명된 제한을 정의한다.

적용 전
Guided Access의 교육 앱: 교사가 수업 중 "공유" 버튼과 외부 링크를 비활성화하고 싶지만 Guided Access가 터치/모션/시간 제한만 제공한다. 교사가 버튼을 수동으로 숨겨 이를 우회한다.
→
적용 후
커스텀 제한 "공유"와 "외부 링크"가 Guided Access UI에 나타난다. 교사가 세션마다 토글한다 — 앱이 제한 상태를 읽고 관련 컨트롤을 그에 따라 숨긴다.
제한 읽기 — 실시간 제한 상태 확인 UIAccessibility.guidedAccessRestriction(withIdentifier: "com.app.sharing")

커스텀 제한을 `UIGuidedAccessRestrictionDelegate`로 등록한 후, 이 메서드로 각 제한의 실시간 상태를 읽는다. `.allow` 또는 `.deny`를 반환한다. 제한된 기능이 사용되는 모든 곳에서 호출한다 — 시작 시뿐만 아니라 — 감독 기기에서 세션이 활성화된 동안 제한이 변경될 수 있기 때문이다.

적용 전
앱이 제한 상태에 따라 시작 시에만 공유 버튼을 비활성화한다. 감독 기기의 MDM 프로파일이 세션 중간에 제한을 업데이트한다 — 공유 버튼이 나머지 세션 동안 활성화된 채로 유지되어 의도를 우회한다.
→
적용 후
공유 버튼의 가시성이 모든 `viewWillAppear`에서 `UIAccessibility.guidedAccessRestriction(withIdentifier: "com.app.sharing")`으로 계산된다. MDM 제한 변경이 다음 화면 표시 시 즉시 적용된다.

Assistive Access
문제 정의

표준 iOS UI는 밀도가 높고 복잡하다 — 작은 아이콘, 다단계 탐색, 모호한 제스처. 지적 장애·다운 증후군·치매가 있는 사용자에게 이 복잡성은 독립적인 기기 사용을 불가능하게 만드는 장벽이다.

솔루션

Assistive Access가 활성화되면 앱이 단순화된 전체 화면 모드로 실행된다. `UIAccessibility.isAssistiveTouchRunning`(간접 프록시)으로 감지하거나 `UIAccessibilityAssistiveAccessIsEnabled` 알림을 확인한다. 간소화된 레이아웃 제공: 큰 버튼, 최소 텍스트, 단계별 흐름.

인지 접근성 단순화 모드 UIAccessibility.isAssistiveTouchRunning

쇼핑·뱅킹·건강 앱처럼 핵심 흐름이 양식·탐색 메뉴·다단계 마법사를 포함하는 모든 소비자 앱. 사용자나 보호자가 큰 버튼 모드를 활성화할 수 있는 설정 플래그를 제공한다.

적용 전
쇼핑 앱: 3탭 하단 바, 작은 상품 그리드, 다단계 결제. 지적 장애 사용자에게 각 화면이 인지적으로 압도적이다.
→
적용 후
Assistive Access 모드: 전체 화면 상품 카드, 큰 "장바구니에 추가" 버튼, 한 단계 확인. 동일한 구매 흐름, 급격히 줄어든 복잡성.
Assistive Access 토글 감지 NotificationCenter.default.addObserver(forName: UIAccessibility.assistiveTouchStatusDidChangeNotification, ...)

앱이 실행 중인 동안 Assistive Access가 토글될 때 동적으로 반응하고 싶은 앱(드물지만 감독 기기에서 가능). 앱 재시작 없이 단순화된 레이아웃으로 UI를 재구축한다.

적용 전
앱이 시작 시에만 Assistive Access 상태를 확인한다. 앱이 열려 있는 동안 보호자가 MDM을 통해 원격으로 Assistive Access를 활성화한다 — 사용자가 강제 종료하고 다시 시작할 때까지 복잡한 UI가 유지된다.
→
적용 후
assistiveTouchStatusDidChangeNotification을 관찰하면 rebuildUI()가 동적으로 트리거된다. MDM 명령 후 몇 초 내에 단순화된 레이아웃이 나타난다 — 앱 재시작 불필요.
보호자 제어 앱 내 단순 모드 @AppStorage("simplifiedMode") var simplifiedMode: Bool = false

보호자나 가족이 인지적 필요가 있는 사람을 위해 기기를 설정하는 소비자 앱(뱅킹, 쇼핑, 건강). 시스템 Assistive Access에 의존하지 않고 보호자가 활성화할 수 있는 Face ID 또는 패스코드로 보호된 앱 내 "단순 모드" 토글을 설정에 제공한다.

적용 전
보호자가 치매가 있는 노부모를 위해 뱅킹 앱을 단순화하고 싶다. 시스템 Assistive Access가 전체 OS를 변경한다 — 너무 강압적이다. 보호자에게 앱 수준의 옵션이 없다.
→
적용 후
설정 → 단순 모드(패스코드 게이트)가 3개의 큰 버튼이 있는 레이아웃으로 전환: 잔액 확인, 이체, 은행 전화. 보호자가 한 번 설정하면 부모가 자신 있게 독립적으로 앱을 사용한다.

Siri / Vocal Shortcuts
문제 정의

Voice Control 사용자와 운동 장애 사용자는 탐색을 완전히 우회하는 음성 활성화 단축키의 혜택을 받는다. Siri 인텐트 도네이션 없이는 주요 앱 동작이 사용자가 도달하기 어려울 수 있는 계층을 통해 여러 번 탭해야 한다.

솔루션

자주 사용하는 동작에 `INShortcut` 객체를 도네이션한다. 사용자가 Siri 설정에서 커스텀 구문을 지정하면 이를 말하는 것만으로 동작이 바로 실행된다 — 앱 탐색 불필요. `INVoiceShortcutCenter`가 등록된 단축키를 관리한다.

음성 호출을 위한 동작 도네이션 INInteraction(intent: intent, response: nil).donate()

사용자가 자주 수행하고 탐색을 건너뛰는 것이 도움이 되는 모든 동작 — 운동 시작, 단골 커피 주문, 사전 설정 메시지 전송. 한 번만이 아니라 성공적인 모든 사용자 상호작용 시 도네이션한다.

적용 전
운동 시작: 앱 열기 → 오늘 탭 → 달리기 탭 → 시작 탭. 손 이동성이 제한된 사용자에게 4번의 탭과 3단계 탐색은 상당한 장벽이다.
→
적용 후
인텐트 도네이션 후, 사용자가 Siri 설정에서 "달리기 시작"을 추가한다. 이를 말하면 즉시 운동이 시작된다 — 탐색 탭 0회, 음성 명령 1회.
NSUserActivity 경량 인텐트 도네이션 let activity = NSUserActivity(activityType: "com.app.viewRecipe"); view.userActivity = activity

매개변수가 필요 없는 탐색 및 콘텐츠 보기 동작 — 기사 읽기, 특정 화면 열기, 연락처 보기. NSUserActivity는 전체 INIntent 도네이션보다 간단하며 Spotlight와 Siri의 단축키 제안에 충분하다.

적용 전
사용자가 매일 "아침 스무디" 레시피를 연다. 앱이 어떤 활동도 도네이션하지 않아 Siri가 제안하지 않는다 — 사용자가 매번 앱 → 레시피 → 스무디 → 아침으로 탐색해야 한다.
→
적용 후
레시피 화면이 나타날 때 NSUserActivity가 도네이션된다. 며칠 후 Siri가 사용자의 습관적인 시간에 "아침 스무디"를 선제적으로 제안한다 — 잠금 화면에서 한 번 탭.
앱 내 음성 단축키 등록 버튼 INUIAddVoiceShortcutButton(shortcut: INShortcut(intent: intent))

인텐트를 도네이션한 후 앱 내에 직접 "Siri에 추가" 버튼을 노출해 사용자가 Siri 설정으로 이동하지 않고 커스텀 구문을 지정할 수 있다. INUIAddVoiceShortcutButton이 전체 등록 시트를 기본적으로 처리한다.

적용 전
앱이 단축키를 도네이션하지만 사용 방법을 사용자에게 알리지 않는다. 음성 활성화에서 가장 많은 혜택을 받을 운동 장애 사용자가 기능을 발견하지 못한다.
→
적용 후
"Siri에 추가" 버튼이 관련 화면에 나타난다. 탭하면 시스템 등록 시트가 표시된다 — 사용자가 "달리기 시작"을 녹음하면 단축키가 활성화된다. 발견과 등록에 10초 미만이 걸린다.

Eye Tracking & Head Pointer
문제 정의

마비·ALS·심각한 운동 장애로 손을 전혀 사용할 수 없는 사용자는 시선 추적이나 머리 움직임으로 iPadOS를 조작한다. 커스텀 스크롤 뷰, 게임 인터페이스, 드래그 타겟이 포인터 기반 입력에 응답하지 않는 경우가 많다.

솔루션

시선 추적과 헤드 포인터는 표준 포인터 이벤트 시스템을 통해 전달된다. 인터랙티브 요소가 `UIPointerInteraction`에 응답하고 `UIHoverGestureRecognizer`를 통해 드웰 클릭(일정 시간 호버로 활성화)이 작동하도록 보장한다.

드웰 클릭 영역 활성화 UIHoverGestureRecognizer(target: self, action: #selector(hover(_:)))

iPad 앱의 모든 커스텀 인터랙티브 요소. 표준 SwiftUI 버튼과 UIKit UIButton은 이미 응답한다 — 제스처 전용 인터랙션이나 캔버스로 그린 UI에 명시적인 포인터 지원이 필요하다.

적용 전
미술 앱의 커스텀 드로잉 캔버스가 `touchesBegan`만 처리한다. 시선 추적이 터치가 아닌 포인터 이벤트를 전송한다 — 캔버스가 시선으로 완전히 조작 불가능하다.
→
적용 후
UIHoverGestureRecognizer를 추가하면 시선 드웰로 드로잉 스트로크와 도구 선택이 트리거된다. 사용자가 눈 움직임만으로 작품을 만든다.
시선 추적 가능 타겟 UIPointerInteraction(delegate: self)

시선 추적이나 헤드 포인터로 사용될 수 있는 iPad 앱의 모든 탭 가능 요소. 시선 추적이 포인터 이벤트를 전송한다 — UIPointerInteraction이 없는 뷰는 호버 피드백을 주지 않아 드웰 활성화 전에 시선이 어디 착지하는지 알 수 없다.

적용 전
음악 앱의 20개 악기 버튼 그리드. 시선 추적 사용자가 버튼 위를 호버한다 — 시각적 반응 없음. 맹목적으로 드웰 클릭해야 해 자주 잘못된 악기를 활성화한다.
→
적용 후
각 버튼에 .lift 효과가 있는 UIPointerInteraction이 있다. 시선 호버 시 버튼이 켜진다 — 드웰 전에 어떤 요소가 활성화될지 명확한 확인을 본다.
시선 정확도를 위한 최소 크기 .frame(minWidth: 60, minHeight: 60)

심각한 운동 장애가 있어 시선 추적만 의존하는 사용자를 위해 설계된 앱. 시선 정확도는 약 2–3°(10인치 iPad에서 ~30–40pt) — Apple의 44pt 최소값이 시선 전용 인터페이스에 부족하다. 주요 동작에 60–80pt를 목표로 한다.

적용 전
시선 조작 AAC 앱의 44×44pt 버튼. 사용자의 자연스러운 눈 떨림이 정확한 타겟팅을 불안정하게 만든다 — 중요한 대화 중 인접 버튼을 눌러 잘못된 단어를 트리거한다.
→
적용 후
주요 구문 버튼이 8pt 간격으로 72×72pt로 확장된다. 더 큰 타겟이 자연스러운 눈 떨림을 흡수한다 — 선택 정확도가 ~65%에서 ~95%로 향상되어 의사소통 보조기가 진정으로 신뢰할 수 있게 된다.
