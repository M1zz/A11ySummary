피쳐별 레시피

"VoiceOver를 지원하려면 정확히 무엇을 해야 하는가." 27개 피쳐를 각각 잃는 사용자 → 최소 구현 → 확장 → 자주 빠뜨리는 것 → 검증 5단으로 풀었습니다.

27 Vision 피쳐 레시피

"이 피쳐를 지원하려면 정확히 무엇을 해야 하는가" 관점에서 정리. 각 피쳐는 잃는 사용자 → 최소 구현 → 확장 → 자주 빠뜨리는 것 → 검증 순으로 읽으면 된다.

VoiceOver

Vision

화면 위 모든 요소를 음성으로 읽어 주는 스크린 리더. 잃는 사용자. 실명·저시력 사용자 전체, 그리고 운전·취사 등 화면을 보기 어려운 상황의 모든 사용자.

최소 구현

// 모든 인터랙티브 요소에
.accessibilityLabel("좋아요")
.accessibilityAddTraits(.isButton)

// 상태가 있다면
.accessibilityValue(isLiked ? "눌림" : "안 눌림")

// 장식용 이미지·구분선은 숨김
.accessibilityHidden(true)

// 의미 단위로 묶기 ("24도 맑음" → 하나로)
.accessibilityElement(children: .combine)

확장 — 풍부한 경험

// 스와이프-삭제의 음성 대체 (필수)
.accessibilityAction(named: "삭제") { delete() }
.accessibilityAction(.magicTap) { togglePlay() }

// 커스텀 그린 차트는 의미를 텍스트로
ChartView()
    .accessibilityRepresentation { Text("1분기 매출 12% 상승") }

// 페이지 변경 / 긴급 알림
AccessibilityNotification.ScreenChanged().post()
var msg = AttributedString("결제 실패")
msg.accessibilitySpeechAnnouncementPriority = .high
AccessibilityNotification.Announcement(msg).post()

// 헤딩 · 정렬 · 커스텀 로터
.accessibilityHeading(.h1)
.accessibilitySortPriority(100)
.accessibilityRotor("중요", entries: starred, entryID: \.id) { Text($0.title) }
자주 빠뜨리는 것 스와이프-삭제 같은 제스처에 .accessibilityAction을 함께 달지 않으면 VoiceOver 사용자는 기능 자체에 접근할 수 없다. Canvas / UIViewRepresentable로 그린 커스텀 UI는 .accessibilityRepresentation 없이는 깡통으로 들린다. 아이콘만 있는 버튼에 .accessibilityLabel 누락도 단골 누락.

검증

Settings → Accessibility → VoiceOver를 켜고, 한 손가락으로 좌→우 스와이프하며 모든 요소가 의미 있는 이름과 상태로 읽히는지. 두 손가락 회전으로 커스텀 로터가 나타나는지. 두 손가락 더블탭(Magic Tap)으로 핵심 동작이 실행되는지.

Dynamic Type — Larger Text

Vision

사용자가 시스템에서 정한 글자 크기. 잃는 사용자. 노안·저시력·고령 사용자 + 작은 글자가 피로한 모든 사용자.

최소 구현

Text("본문").font(.body)                                  // 자동 스케일

// 커스텀 폰트는 relativeTo: 사용
Text("본문").font(.custom("Fraunces", size: 17, relativeTo: .body))

확장 — 접근성 크기 대응

@Environment(\.dynamicTypeSize) var typeSize
@ScaledMetric var iconSize: CGFloat = 24

// 가로 → 세로 레이아웃 전환
if typeSize.isAccessibilitySize {
    VStack { Avatar(); Text(name); Text(role) }
} else {
    HStack { Avatar(); VStack { Text(name); Text(role) } }
}

// 또는 자동 적응
ViewThatFits {
    HStack { Toolbar() }
    VStack { Toolbar() }
}
자주 빠뜨리는 것 고정 size만 사용하는 폰트, 고정 height·lineLimit(1)로 인한 클리핑, 접근성 크기에서 깨지는 가로 카드. minimumScaleFactor를 너무 일찍 쓰면 접근성 크기 자체를 무력화한다.

검증

Settings → Accessibility → Display & Text Size → Larger Text의 슬라이더를 최대(AX5)까지 올리고 모든 화면을 둘러본다. 잘리는 텍스트·터치 어려워진 버튼·겹치는 컨텐츠를 확인.

Bold Text

Vision

시스템 폰트를 항상 굵게. 잃는 사용자. 저시력, 콘트라스트 민감 사용자.

@Environment(\.legibilityWeight) var weight    // .regular | .bold

// 시스템 폰트 — 자동 적용, 별도 작업 불필요
Text("본문").font(.body)

// 커스텀 폰트 — 분기 필수
let name = weight == .bold ? "Fraunces-Bold" : "Fraunces-Regular"
Text("본문").font(.custom(name, size: 17, relativeTo: .body))

Increase Contrast

Vision

대비를 시스템 차원에서 높임. 잃는 사용자. 저시력, 노안, 야외 사용 환경의 모든 사용자.

@Environment(\.colorSchemeContrast) var contrast   // .standard | .increased

// 방법 1: Asset Catalog의 "High Contrast" 변형 — 자동 적용
// 방법 2: 코드 분기
Text("부가 설명")
    .foregroundStyle(contrast == .increased ? Color.primary : Color.secondary)
자주 빠뜨리는 것 회색 톤(secondary, tertiary)을 placeholder가 아닌 본문에 사용. 본문은 WCAG AA 기준 4.5:1, 대형 텍스트는 3:1을 항상 만족해야 한다. Increase Contrast가 켜진 환경에서는 한 단계 더.

Reduce Transparency

Vision

블러·반투명을 솔리드로. 잃는 사용자. 인지 부담, 저시력.

@Environment(\.accessibilityReduceTransparency) var reduce

Rectangle()
    .fill(reduce ? Color(.systemBackground) : .ultraThinMaterial)

Differentiate Without Color

Vision

색만으로 정보를 전달하지 않기. 잃는 사용자. 색맹(남성 8%·여성 0.5%), 저시력, 흑백 출력·캡처 환경.

최소 구현 — 색 외 단서 항상 동반

// 색 + 아이콘 + 텍스트 동시 사용
HStack {
    Image(systemName: status.isError ? "xmark.circle.fill" : "checkmark.circle.fill")
    Text(status.message)
}
.foregroundStyle(status.isError ? .red : .green)

확장 — 환경값 켜졌을 때 더 강한 단서

@Environment(\.accessibilityDifferentiateWithoutColor) var diff

ZStack {
    Circle().fill(status.color)
    if diff {
        Image(systemName: status.symbol)          // 켜졌을 때만 아이콘 추가
            .foregroundStyle(.white)
    }
}
자주 빠뜨리는 것 빨강·녹색만으로 상태를 구분하는 그래프, 색 점만으로 "안 읽음"을 표시하는 리스트, 색만으로 "선택됨"을 표시하는 칩. 모두 추가 단서가 필요하다.

Smart / Classic Invert

Vision

전체 색 반전. Smart Invert는 사진·미디어를 자동 제외. 잃는 사용자. 광 민감성, 어두운 환경 선호.

// 의미가 색에 묶인 이미지(사진·로고·아이콘)는 반전 제외
Image("product-photo")
    .accessibilityIgnoresInvertColors()

Image("brand-logo")
    .accessibilityIgnoresInvertColors()

Button Shapes

Vision

버튼처럼 보이는 명확한 경계 강제. 잃는 사용자. 인지 보조 필요, "이게 누를 수 있는 건가" 헷갈리는 모든 사용자.

@Environment(\.accessibilityShowButtonShapes) var shapes

// 시스템 Button — 자동
Button("저장") { save() }

// 텍스트 링크형 버튼은 분기 필요
Text("건너뛰기")
    .padding(.horizontal, 12).padding(.vertical, 6)
    .overlay(
        shapes ? AnyView(Capsule().stroke(.tint, lineWidth: 1)) : AnyView(EmptyView())
    )
    .onTapGesture { skip() }

Large Content Viewer (Zoom 보조)

Vision

작은 컨트롤을 탭 앤 홀드로 크게 미리보기. 잃는 사용자. Zoom 사용자, 저시력, 작은 탭바 아이콘이 헷갈리는 모든 사용자.

Button { ... } label: { Image(systemName: "heart") }
.accessibilityShowsLargeContentViewer {
    Label("좋아요", systemImage: "heart.fill")
}

// 인자 없이 호출하면 label/value가 그대로 사용됨
.accessibilityShowsLargeContentViewer()

시스템 전용 — 앱 구현 없음

Zoom · Magnifier · Color Filters · Hover Typing · Spoken Content · Audio Descriptions(자동 트랙). OS가 우리 앱 화면 위에서 동작하므로 별도 SwiftUI 구현이 없다. 우리가 할 일은 (1) 위 피쳐들을 잘 지원해서 줌 의존도 자체를 낮추고, (2) 영상에는 caption + audio description 트랙을 임베드해 두는 것.

28 Hearing 피쳐 레시피

청각 카테고리는 앱 측 구현 포인트가 적다. 두 가지만 챙기면 된다 — 자막 트랙과 Music Haptics.

Closed Captions / Live Captions 기여

Hearing

영상·오디오의 자막. 잃는 사용자. 청각 장애·난청, 소음 환경, 외국어 학습자.

최소 구현

import AVKit

// caption 트랙이 임베드되어 있으면 VideoPlayer가 자동 처리
VideoPlayer(player: AVPlayer(url: videoURL))

// 자체 자막 텍스트를 그릴 때는 .accessibilityLabel 동기화
SubtitleView(text: currentLine)
    .accessibilityLabel(currentLine)              // VoiceOver도 읽음
자주 빠뜨리는 것 자막을 영상에 burn-in(이미지로 굽기)으로 그려 넣으면 시스템 Live Captions가 읽지 못하고 사용자가 자막을 끄거나 크기를 조절할 수도 없다. 반드시 별도 caption 트랙(SRT·WebVTT·CMAF)으로 임베드.

Music Haptics 통합

Hearing

자체 음악 플레이어에서 시스템 Music Haptics 트랙을 재생. 잃는 사용자. 청각 장애·난청, 그리고 햅틱으로 음악을 즐기는 모든 사용자.

import MusicHaptics

struct PlayerView: View {
    let song: Song
    @State var mode: MusicHapticsMode = .full

    var body: some View {
        VStack {
            PlayPauseButton()
            Picker("햅틱", selection: $mode) {
                Text("전체").tag(MusicHapticsMode.full)
                Text("보컬만").tag(MusicHapticsMode.vocalsOnly)
            }
        }
        .task(id: song.id) { await start() }
    }

    func start() async {
        let mgr = MusicHapticsManager.shared
        guard await mgr.requestAuthorization() == .authorized,
              await mgr.isHapticTrackAvailable(forSongID: song.id) else { return }
        try? await mgr.play(songID: song.id, mode: mode, intensity: 0.8)
    }
}

시스템 전용 — 앱 구현 없음

Live Captions(시스템 와이드) · Sound Recognition · Name Recognition · Live Listen · MFi Hearing Devices · Background Sounds · Mono Audio · RTT. 우리 앱이 할 일은 (1) 정확한 AVAudioSession 카테고리 설정으로 시스템이 오디오 맥락을 이해하게 돕고, (2) 알림음·신호음에 항상 시각적 등가물을 동반.

29 Mobility 피쳐 레시피

손·팔의 자유로운 사용이 어려운 사용자들. 입력 수단이 키보드·스위치·시선·음성 등 다층으로 갈라지므로, 같은 동작에 여러 경로를 함께 제공하는 것이 핵심이다.

Switch Control

Mobility

외부 스위치(또는 카메라·머리·BCI)로 UI를 스캐닝하며 조작. 잃는 사용자. ALS, 척수 손상, 중증 운동 장애.

최소 구현

// VoiceOver용 구현(label/traits/actions)이 그대로 적용된다
.accessibilityLabel("삭제")
.accessibilityAddTraits(.isButton)
.accessibilityAction(named: "답장") { reply() }

확장 — 스캐닝 효율

// 의미 있는 그룹 단위로 묶기 (한 번에 점프할 수 있게)
VStack { Avatar(); Name(); Role(); Actions() }
    .accessibilityElement(children: .contain)
    .accessibilityRespondsToUserInteraction(true)

// 보조 정보를 customContent로 — 기본 라벨을 짧게 유지
.accessibilityCustomContent("우선순위", item.priority.name)
자주 빠뜨리는 것 셀의 핵심 동작이 swipe-only(예: 좌우 스와이프로만 삭제 가능)인 경우 Switch Control 사용자는 영원히 그 기능에 도달할 수 없다. 반드시 .accessibilityAction으로 동일 동작 제공.

검증

Settings → Accessibility → Switch Control에서 "Camera" 스위치(머리 움직임 좌우)를 등록해 켜면 추가 하드웨어 없이 테스트할 수 있다.

Voice Control

Mobility

음성 명령으로 UI 조작. 잃는 사용자. 손 사용이 어렵거나 음성을 선호하는 사용자, 핸즈프리 환경.

최소 구현

// 화면에 보이는 텍스트 라벨이 1순위 명령어가 된다
Button("삭제") { delete() }

// 아이콘 버튼은 반드시 accessibilityLabel
Button { delete() } label: { Image(systemName: "trash") }
    .accessibilityLabel("삭제")

확장 — 별칭과 다국어

.accessibilityInputLabels(["삭제", "휴지통", "trash", "delete"])
자주 빠뜨리는 것 한 화면에 같은 이름의 버튼이 여러 개 있으면 시스템이 "1, 2, 3 번 보기"로 강제 전환되어 사용성이 떨어진다. accessibilityInputLabels로 구분되는 별칭을 줘야 한다.

검증

Settings → Accessibility → Voice Control 켜기. "Show names"로 라벨 오버레이를, "Show numbers"로 그리드 모드를 확인.

AssistiveTouch · Touch Accommodations

Mobility

가상 버튼으로 시스템 동작 대체, 또는 터치 감도·홀드 시간 조정. 잃는 사용자. 미세 동작·정밀 제어 어려움, 떨림·경직.

Button { ... } label: { Image(systemName: "xmark") }
    .frame(minWidth: 44, minHeight: 44)         // HIG 최소 hit target
    .contentShape(Rectangle())                        // 시각 그대로, 히트만 확장
자주 빠뜨리는 것 빠른 연속 탭(double-tap), 정밀 드래그(슬라이더), 동시 멀티터치(핀치 줌)에만 의존하는 인터랙션. 모두 단일 탭 대체 경로(.accessibilityAction, +/− 버튼, 키보드 단축키)와 함께 제공.

Eye Tracking · Head Tracking

Mobility

시선·머리 움직임으로 포인터 제어. 잃는 사용자. 사지마비, 중증 운동 장애.

InteractiveTile()
    .focusable(true)
    .hoverEffect(.automatic)                          // 응시 피드백 — 거의 공짜
    .frame(minWidth: 60, minHeight: 60)         // 시선 떨림 고려
자주 빠뜨리는 것 호버 시 시각 피드백이 없으면 사용자는 "지금 어디를 보고 있는지" 알 수 없다. .hoverEffect는 거의 모든 인터랙티브 요소에 거의 공짜로 추가 가능하다.

Full Keyboard Access

Mobility

외장 키보드만으로 모든 화면 탐색·조작. 잃는 사용자. 키보드 전용 사용자, 그리고 외장 키보드를 스위치로 매핑해 쓰는 모든 Switch Control 사용자.

최소 구현

enum Field { case id, pw }
@FocusState var field: Field?

Form {
    TextField("ID", text: $id).focused($field, equals: .id)
    SecureField("PW", text: $pw).focused($field, equals: .pw)
    Button("로그인") { login() }
        .keyboardShortcut(.return, modifiers: [])
}
.defaultFocus($field, .id)

확장 — 키 직접 처리

.onKeyPress(.escape) { dismiss(); return .handled }
.onKeyPress(characters: ["j"]) { selectNext(); return .handled }
.focusSection()                                       // macOS/tvOS/visionOS 그룹
두 포커스 동기화 @FocusState(키보드)와 @AccessibilityFocusState(VoiceOver / Switch Control)는 별개 시스템이다. 에러 발생 시 둘 다 에러 메시지로 이동시켜야 키보드 사용자와 VoiceOver 사용자 모두에게 즉시 인지된다.

Vocal Shortcuts · Back Tap

Mobility

사용자가 녹음한 짧은 발성, 또는 기기 뒷면 탭으로 액션 실행. 잃는 사용자. 다양한 모빌리티, 그리고 단축 진입을 원하는 모든 사용자.

구현 — App Intent로 액션 노출

import AppIntents

struct StartTimerIntent: AppIntent {
    static let title: LocalizedStringResource = "5분 타이머 시작"
    static let isDiscoverable = true

    func perform() async throws -> some IntentResult {
        await TimerService.shared.start(minutes: 5)
        return .result()
    }
}

사용자는 Settings → Accessibility → Vocal Shortcuts 또는 Touch → Back Tap에서 이 인텐트를 자신만의 트리거에 바인딩한다. 같은 인텐트가 Siri·Shortcuts·Spotlight·홈 위젯에서도 재사용된다.

30 Speech 피쳐 레시피

발화 보조 카테고리. SwiftUI는 입력 UI만 담당하고 본체는 AVFoundation. AAC(Augmentative and Alternative Communication) 앱을 만들 때 가장 직접적으로 통합된다.

Personal Voice 사용

Speech

사용자가 학습시킨 자신의 합성 음성으로 발화. 잃는 사용자. ALS·후두암 등으로 발화 능력을 잃을 위험이 있거나 이미 잃은 사용자.

최소 구현

import AVFoundation

struct AACView: View {
    @State var text = ""
    @State var myVoice: AVSpeechSynthesisVoice?
    let synth = AVSpeechSynthesizer()

    var body: some View {
        VStack {
            TextField("하고 싶은 말", text: $text, axis: .vertical)
                .font(.system(size: 28))
            Button("말하기") { speak() }
                .keyboardShortcut(.return)
        }
        .task { await setup() }
    }

    func setup() async {
        let status = await withCheckedContinuation { c in
            AVSpeechSynthesizer.requestPersonalVoiceAuthorization { c.resume(returning: $0) }
        }
        if status == .authorized {
            myVoice = AVSpeechSynthesisVoice.speechVoices()
                .first { $0.voiceTraits.contains(.isPersonalVoice) }
        }
    }

    func speak() {
        let u = AVSpeechUtterance(string: text)
        u.voice = myVoice ?? AVSpeechSynthesisVoice(language: "ko-KR")
        synth.speak(u)
    }
}
필수 설정 Info.plist에 NSPersonalVoiceUsageDescription이 없으면 인증 요청 자체가 실패한다. Apple은 AAC가 아닌 일반 TTS 목적의 Personal Voice 사용을 승인하지 않는다.

Live Speech 자체 구현

Speech

시스템 Live Speech와 별개로, 우리 앱이 자체 AAC 인터페이스를 제공할 때. 잃는 사용자. 일시적 발화 보조가 필요한 모든 사용자(수술 후 회복, 후두염 등).

FavoritePhrasesView()                              // 즐겨찾기 문장 그리드
    .accessibilityShowsLargeContentViewer()        // 시력 보조도 함께

Button(phrase) {
    let u = AVSpeechUtterance(string: phrase)
    u.voice = AVSpeechSynthesisVoice(language: "ko-KR")
    synth.speak(u)
}
.font(.system(size: 24, weight: .medium))
.frame(maxWidth: .infinity, minHeight: 80)

시스템 전용 — 앱 구현 없음

Listen for Atypical Speech · Hold for Siri. SFSpeechRecognizer와 시스템 Siri가 자동으로 사용자 설정을 따른다. STT를 정직하게 사용하기만 하면 자동 혜택.

31 Cognitive 피쳐 레시피

인지 부담·전정 자극·읽기 어려움을 줄이고 단순화하는 카테고리. 환경 변수 분기와 App 구조 자체에서 결정된다.

Assistive Access

Cognitive

큰 버튼·단일 작업·하이콘트라스트의 단순화된 모드. 잃는 사용자. 지적·발달 장애, 인지 부담이 큰 사용자, 그리고 일시적으로 단순화된 인터페이스가 도움되는 모든 상황.

최소 구현

@main
struct MyApp: App {
    var body: some Scene {
        WindowGroup { ContentView() }              // 일반 모드

        AssistiveAccessScene {                      // 단순화 별도 Scene
            SimpleHomeView()
        }
    }
}

struct SimpleHomeView: View {
    var body: some View {
        VStack(spacing: 24) {
            BigActionButton(title: "전화 걸기", icon: "phone.fill") { ... }
            BigActionButton(title: "사진 보기", icon: "photo.fill") { ... }
        }
        .font(.system(size: 32, weight: .semibold))
        .padding(24)
    }
}
설계 원칙 Assistive Access UI는 (1) 한 화면 한 작업, (2) 화면 폭의 1/3 이상의 큰 컨트롤, (3) 단순한 동사("전화 걸기"), (4) 항상 보이는 되돌리기·종료. 모달·딥 네비게이션·미세 제스처는 모두 금지.

Reduce Motion

Cognitive

애니메이션·모션 축소. 잃는 사용자. 전정 장애, 멀미, ADHD·자폐로 인한 자극 민감, 광 민감성 발작 위험.

@Environment(\.accessibilityReduceMotion) var reduce

// 1. 단일 애니메이션 게이팅
.scaleEffect(scale)
.animation(reduce ? nil : .spring, value: scale)

// 2. 트랜잭션 단위 비활성화
var txn = Transaction()
txn.disablesAnimations = reduce
withTransaction(txn) { state.toggle() }

// 3. 자체 모션도 대체
if reduce {
    Image("hero")                                  // 정적
} else {
    AnimatedHero()                                  // 패럴랙스·바운스
}
자주 빠뜨리는 것 무분별한 .transition(.slide)·.matchedGeometryEffect·자동 캐러셀·패럴랙스 스크롤. 모두 환경값 분기 필요.

Cross-Fade Transitions

Cognitive

슬라이드 전환을 페이드로. 잃는 사용자. 전정 자극에 민감한 사용자(Reduce Motion과 별개의 토글).

@Environment(\.accessibilityPrefersCrossFadeTransitions) var fade

// SwiftUI NavigationStack은 자동 — 직접 구현한 전환만 분기 필요
.transition(fade ? .opacity : .slide)

Animated Images 정책

Cognitive

GIF·Lottie 자동 재생 여부. 잃는 사용자. 광 민감성, 집중력 이슈, 인지 부담.

@Environment(\.accessibilityPlayAnimatedImages) var play

GIFView(url: gif, isAnimating: play)
LottieView(animation: anim, isPlaying: play)

Guided Access 통합 (UIKit 브릿지)

Cognitive

특정 앱·기능에 잠금. 잃는 사용자. 자폐 스펙트럼, ADHD, 또는 학부모·치료사가 학생·환자에게 한정된 환경을 제공해야 하는 모든 경우.

class AppDelegate: NSObject, UIApplicationDelegate, UIGuidedAccessRestrictionDelegate {
    func guidedAccessRestrictionIdentifiers() -> [String]? {
        ["com.app.chat", "com.app.purchase"]
    }
    func textForGuidedAccessRestriction(withIdentifier id: String) -> String? {
        id == "com.app.chat" ? "채팅 사용 허용" : "인앱 구매 허용"
    }
    func guidedAccessRestriction(withIdentifier id: String,
                                  didChange state: UIAccessibility.GuidedAccessRestrictionState) {
        Restrictions.shared.apply(id: id, state: state)
    }
}

@main struct MyApp: App {
    @UIApplicationDelegateAdaptor(AppDelegate.self) var delegate
    var body: some Scene { WindowGroup { ContentView() } }
}

Accessibility Reader 친화

Cognitive

시스템 와이드 리더 모드(2025). 잃는 사용자. 난독증, 읽기 학습 중인 사용자, 노안.

Text(article.body)
    .accessibilityTextContentType(.narrative)     // 본문 컨텍스트 명시

Text(code)
    .accessibilityTextContentType(.sourceCode)

본문을 Text로 깔끔히 노출하고 컨텐츠 타입을 명시하면 Accessibility Reader가 자동으로 사용자 환경에 맞춰 재조판한다. 텍스트를 이미지로 굽거나 그래픽 위에 그리지 않는 것이 가장 중요한 원칙.

시스템 전용 — 앱 구현 없음

Vehicle Motion Cues · Background Sounds(시스템). 모션이 강한 화면에서 Reduce Motion을 잘 따르고, 모션 발생 영역에 시각적 닻을 함께 두는 것으로 기여한다.
