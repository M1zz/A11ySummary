기능 × SwiftUI

"Vision 카테고리에는 이런 SwiftUI API들이 있다" — 5개 카테고리별 매핑. 그리고 "이 모디파이어를 안 쓰면 누가 떨어져 나가는가" 관점의 API → 기능 역인덱스.

19 개요 — 기능 × API 관계

같은 SwiftUI 모디파이어가 여러 사용자 기능을 동시에 떠받친다. .accessibilityLabel 하나만 빠뜨려도 VoiceOver · Voice Control · Switch Control · Assistive Access 사용자가 동시에 떨어져 나간다. 이 파트는 "기능 → 어떤 API"와 "API → 어떤 기능"을 양방향으로 정리한다.

읽는 법 카테고리별로 사용자 기능마다 필요한 SwiftUI 모디파이어·환경 변수·이벤트를 묶어 나열한다. 같은 API가 여러 곳에 등장하는 중복은 의도된 것이며, 그것이 바로 핵심 통찰이다.

20 Vision × SwiftUI

시각 카테고리가 SwiftUI 접근성 API의 80%를 차지한다. VoiceOver 관련 모디파이어가 가장 많고, 다른 시각 기능들이 그 위에서 재사용된다.

VoiceOver — 스크린 리더

Vision

의미·구조 부여

.accessibilityLabel("좋아요")
없을 때
하트 모양 아이콘만 있는 버튼은 VoiceOver가 heart fill 같은 시스템 아이콘 이름을 영어로 읽거나, 라벨이 정말 없으면 그냥 침묵한다.
적용 후
포커스되는 순간 좋아요라고 또렷이 발화. Traits이 함께 있으면 좋아요. 버튼.
언제
텍스트 없이 아이콘·이미지만 보여 주는 모든 인터랙티브 요소. Button("저장")처럼 라벨이 들어 있는 표준 컴포넌트는 자동으로 잡힌다.
.accessibilityValue("눌림")
없을 때
토글의 현재 상태(켜짐/꺼짐)나 슬라이더의 값을 알 길이 없다. 좋아요. 버튼.만 들린다.
적용 후
좋아요. 눌림. 버튼. — 현재 상태가 라벨 직후에 함께 발화된다.
언제
슬라이더 값, 토글 상태, 텍스트필드 내용, 진행률 등 "지금의 값/상태"가 있는 모든 요소. Toggle·Slider·TextField 같은 표준 컴포넌트는 자동.
.accessibilityHint("두 번 탭하면 좋아요")
없을 때
사용자는 이 요소를 두 번 탭했을 때 무엇이 일어날지 모른다.
적용 후
포커스 후 약 1초 뒤에 두 번 탭하면 좋아요가 추가로 발화된다.
주의
사용자가 Settings에서 hint 발화를 꺼 두었다면 들리지 않는다. 핵심 의미는 반드시 label에 담고, hint는 보조 정보로만. "두 번 탭하면 열림" 같은 자명한 hint는 오히려 소음.
.accessibilityAddTraits(.isButton)
없을 때
Text나 HStack에 .onTapGesture만 붙인 요소는 그저 평범한 텍스트로 들린다. 사용자는 탭 가능한지 알지 못한다.
적용 후
좋아요. 버튼. — 사용자는 "두 번 탭으로 활성화" 패턴을 인지한다.
다른 trait
.isHeader(헤딩) · .isLink(링크) · .isSelected(선택됨) · .isModal(모달, 형제 차단) · .isToggle(토글) · .updatesFrequently(타이머·라이브 점수처럼 자주 변하는 값) · .startsMediaSession(미디어 트리거 — VoiceOver가 자기 음성을 일시 멈춤) · .playsSound(소리 재생) · .allowsDirectInteraction(피아노·드로잉처럼 직접 터치가 의미를 갖는 영역) · .causesPageTurn(전자책 페이지 전환).
.accessibilityRemoveTraits(.isHeader)
역할
시스템이 자동으로 부여한 trait를 제거. 예: SwiftUI Section header는 자동으로 .isHeader가 되는데, 이것을 단순 시각 그루핑으로 격하시키고 싶을 때.
적용 후
사용자가 헤딩 로터로 점프할 때 그 요소가 후보에서 제외된다.
.accessibilityHeading(.h1)
없을 때
화면 위 모든 텍스트가 평평하게 들린다. 사용자는 페이지의 위계와 구조를 파악할 수 없다.
적용 후
사용자가 VoiceOver 로터를 "헤딩"으로 돌리면 h1 → h2 → h3 사이를 점프하며 빠르게 탐색할 수 있다. 웹 페이지의 H 위계와 같은 개념.
언제
화면이 명확한 위계를 가지는 경우 — 설정 카테고리, 긴 글, 폼 섹션 등.
.accessibilityTextContentType(.sourceCode)
없을 때
func loadUser() { } 같은 코드를 자연어처럼 읽으려 하다 보니 들여쓰기·심볼 발음·줄바꿈을 모두 놓친다. 펑크 로드 유저 파렌처럼 어색하게 들린다.
적용 후
VoiceOver가 코드 모드로 전환되어 들여쓰기 단계를 알리고, 심볼({, ;, =)을 또렷이 발음하며, 줄바꿈을 청각적 단락으로 처리한다.
다른 타입
.console(터미널 출력) · .fileSystem(경로) · .messaging(채팅) · .narrative(본문 산문) · .spreadsheet · .wordProcessing. Accessibility Reader도 이 정보를 함께 활용한다.
.accessibilitySortPriority(100)
없을 때
VoiceOver는 시각 순서(위→아래, 시작→끝)를 그대로 따라 읽는다.
적용 후
같은 컨테이너 내에서 priority 값이 큰 요소가 먼저 읽힌다. 기본값 0.
언제
시각 레이아웃과 청각 우선순위가 다를 때. 예: 상품 카드에서 큰 가격 숫자가 시각적으로 오른쪽 끝에 있지만, 사용자가 가장 먼저 듣고 싶은 정보는 가격일 때.
.accessibilityIdentifier("like-button")
음성 출력
VoiceOver와 무관 — 절대 발화에 사용되지 않는다.
용도
XCUITest 같은 UI 테스트가 요소를 찾는 식별자. app.buttons["like-button"].tap() 같은 테스트 코드에서 사용.
주의
실제 사용자에게는 절대 들리지 않는다. 디버깅·테스트 전용 메타데이터.

그룹화·숨기기·표현 대체

.accessibilityElement(children: .combine)
없을 때
HStack { Image("sun"); Text("24°C"); Text("맑음") }는 세 요소가 따로 읽힌다. 사용자는 한 카드의 정보를 얻으려 스와이프를 세 번 해야 한다.
적용 후
한 요소로 합쳐져 맑음 아이콘. 24도. 맑음.으로 한 번에 읽힌다. 스와이프 한 번에 카드 전체 정보.
변형
.ignore(자식 무시, 컨테이너만 단일 요소로 노출) · .combine(자식 라벨을 합쳐 단일 요소로) · .contain(자식은 그대로 유지, 컨테이너만 추가).
.accessibilityChildren { ChildLabels() }
역할
접근성 자식 트리를 시각 트리와 별개로 직접 제공. 시각적으로는 한 덩어리의 그림이지만 VoiceOver엔 여러 개로 보이게 하고 싶을 때.
적용 후
VoiceOver는 시각 뷰를 무시하고 ChildLabels()의 자식 뷰들을 본다. 시각엔 영향 없음.
예
커스텀 그린 평점 위젯이 시각적으로 단일 별 그래픽이지만, 5개의 별 각각을 VoiceOver에 노출하고 싶을 때.
.accessibilityRepresentation { Text("매출 차트, 12% 상승") }
없을 때
Canvas·UIViewRepresentable로 그린 차트는 그저 이미지로 들리거나 모든 픽셀이 의미 없이 침묵한다.
적용 후
VoiceOver는 시각 뷰를 완전히 무시하고 안의 Text만 본다. 결과: 매출 차트, 12% 상승.
언제
커스텀 그리기로 만든 차트·게이지·복잡한 시각화 — SwiftUI 표준 컴포넌트로 다시 만들기 어려운 경우. 안의 View는 SwiftUI 표준 컴포넌트(Text·Button 등)면 무엇이든 OK.
.accessibilityHidden(true)
없을 때
모든 요소가 트리에 포함된다. 장식용 구분선·배경 이미지·아이콘이 함께 읽혀 소음을 만든다.
적용 후
그 요소(및 자식들)가 VoiceOver 탐색에서 완전히 제외. 시각엔 그대로, 청각에서는 사라진다.
언제
의미 없는 장식 이미지, 텍스트 옆의 보조 그래픽, 텍스트와 동일한 의미가 이미 라벨에 들어간 아이콘 등.
.accessibilityLinkedGroup(id: "row-3", in: namespace)
역할
화면 곳곳에 흩어진 여러 요소가 "같은 그룹"임을 표시.
적용 후
사용자가 한 그룹 안에서 두 손가락 회전 로터의 "Linked Items"로 같은 그룹의 다음/이전 요소로 점프 가능.
예
캐러셀 페이지 인디케이터가 화면 하단에, 본문이 가운데에 있을 때 둘을 같은 linked group으로 묶으면 인디케이터 ↔ 본문 ↔ 다음 인디케이터로 이어 탐색할 수 있다.
.accessibilityLabeledPair(role: .label, id: "name", in: namespace)
역할
폼에서 라벨과 필드의 관계를 명시. role: .label과 .content 한 쌍을 같은 id로 묶는다.
적용 후
사용자가 필드(content)에 포커스하면 자동으로 라벨 텍스트가 함께 발화된다. 이메일. 텍스트 필드. 비어 있음.
언제
SwiftUI Label·TextField의 첫 인자 라벨을 쓰지 않고 시각적으로 떨어진 위치에 라벨을 따로 둔 커스텀 폼.

커스텀 액션

.accessibilityAction(named: "답장") { reply() }
없을 때
스와이프-삭제 같은 제스처에 묶인 동작은 VoiceOver 사용자가 도달할 수 없다 — 기능 자체에 접근 불가.
적용 후
요소에 포커스한 상태에서 한 손가락 위·아래 스와이프로 액션 목록이 등장. 답장 같은 옵션을 선택 후 두 번 탭으로 실행. 액션이 여러 개면 자동으로 액션 로터에 모인다.
언제
스와이프 액션, long-press 메뉴, hover 메뉴 같은 "숨은" 동작 전부.
.accessibilityAction(.magicTap) { togglePlayback() }
매직 탭
두 손가락 더블탭. 시스템 전역에서 "이 앱의 가장 핵심적인 토글"을 의미하는 표준 제스처.
적용 후
화면 어디에서든 두 손가락 더블탭만으로 핸들러가 실행된다 — 핵심 동작을 위해 화면 위에서 버튼을 찾아 헤맬 필요가 없어진다.
관용
음악 앱은 재생/정지, 통화 앱은 받기/끊기, 타이머는 시작/일시정지. 한 앱에 하나만 두는 것이 원칙.
.accessibilityAction(.escape) { dismiss() }
제스처
두 손가락으로 Z자를 그리는 동작. "취소·뒤로·닫기"의 표준 음성 제스처.
적용 후
모달·시트·전체 화면 커버를 빠르게 닫을 수 있다. SwiftUI .sheet·NavigationStack은 자동 처리됨.
언제
직접 만든 모달 오버레이, 커스텀 풀스크린, 자체 라우팅으로 띄운 뷰.
.accessibilityAdjustableAction { dir in ... }
없을 때
커스텀 슬라이더·스테퍼는 VoiceOver 사용자가 값을 조정할 수 없다. 두 번 탭으로 "활성화"는 되지만 그 다음이 없다.
적용 후
요소에 포커스한 상태에서 한 손가락 위·아래 스와이프로 핸들러가 .increment·.decrement로 호출된다. VoiceOver는 자동으로 .adjustable trait을 부여해 조정 가능임을 알린다.
함께
반드시 .accessibilityValue로 현재 값을 발화해야 한다. 음량. 60 퍼센트. 조정 가능.
.accessibilityScrollAction { dir in ... }
제스처
세 손가락 스와이프(상·하·좌·우) — 페이지 단위 스크롤의 표준 음성 제스처.
적용 후
커스텀 페이저·캐러셀에서도 세 손가락 스와이프로 페이지가 넘어간다. 핸들러의 dir로 방향 처리.
언제
SwiftUI ScrollView·TabView는 자동. 직접 만든 페이지 컨테이너에서만 필요.
.accessibilityZoomAction { event in ... }
제스처
핀치 줌의 음성 대체 — 두 손가락 더블탭 앤 홀드 후 위로 드래그(확대) / 아래로 드래그(축소).
적용 후
핸들러의 event.direction(.zoomIn / .zoomOut)으로 줌 인·아웃 처리.
언제
지도·사진 뷰어 등 직접 구현한 핀치 줌이 있는 영역.

23 Speech × SwiftUI

발화 카테고리는 SwiftUI보다 AVFoundation·Speech 프레임워크가 본체. SwiftUI는 인증 UI와 텍스트 입력을 담당한다.

Personal Voice 사용 (AAC 앱)

Speech
import AVFoundation

struct SpeakView: View {
    @State var input = ""
    @State var voice: AVSpeechSynthesisVoice?
    let synth = AVSpeechSynthesizer()

    var body: some View {
        VStack {
            TextField("할 말", text: $input)
            Button("말하기") { speak() }
                .keyboardShortcut(.return)
        }
        .task { await loadVoice() }
    }

    func loadVoice() async {
        let status = await withCheckedContinuation { c in
            AVSpeechSynthesizer.requestPersonalVoiceAuthorization { c.resume(returning: $0) }
        }
        if status == .authorized {
            voice = AVSpeechSynthesisVoice.speechVoices()
                .first { $0.voiceTraits.contains(.isPersonalVoice) }
        }
    }

    func speak() {
        let u = AVSpeechUtterance(string: input)
        u.voice = voice ?? AVSpeechSynthesisVoice(language: "ko-KR")
        synth.speak(u)
    }
}
필수 설정 Info.plist에 NSPersonalVoiceUsageDescription 추가. 보조 통신(AAC) 목적이 아니면 승인되지 않는다.

26 80%를 커버하는 6가지 우선순위

매번 모든 모디파이어를 챙기긴 어렵다. 이 순서대로만 챙겨도 Accessibility Nutrition Labels의 주요 항목 대부분이 살아남는다.

  1. .accessibilityLabel + 적절한 Traits — VoiceOver · Voice Control · Switch Control이 한 번에 살아남는다.
  2. Dynamic Type 대응 — .font(.body) 또는 relativeTo:, typeSize.isAccessibilitySize로 레이아웃 분기.
  3. \.accessibilityReduceMotion 분기 — 멀미·전정 장애·집중력 이슈가 있는 사용자.
  4. \.accessibilityDifferentiateWithoutColor 대응 — 색맹·저시력 사용자에게 색 외 단서 추가.
  5. 44pt 최소 hit target + .contentShape — Mobility 카테고리 전반.
  6. 커스텀 제스처는 반드시 .accessibilityAction으로도 노출 — 스와이프-삭제·핀치-줌 같은 제스처에 의존하면 Switch Control · Voice Control 사용자는 접근 불가가 된다.
